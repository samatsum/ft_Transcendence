# CODING_RULES - cub3D C Coding Rules

この文書を cub3D の C コーディングルールの正本とする。`make check` は CR001〜CR015 を検査し、失敗したら必ず直す。`make audit` は CR001〜CR016 を実行し、CR016 のような助言系も表示する。CR017 以降は人間レビューで確認する。

## レビュー基準

コードレビューでは、単にフォーマットだけでなく次を確認する。

1. モジュールの関係性と粒度
   - 関数・ファイルが単一責任に近い粒度で分割されていること。
   - 高凝集・低結合で、モード固有仕様が common に漏れすぎていないこと。
   - 不要なグローバル状態や外部リンケージを避け、副作用の範囲を小さくすること。

2. 複雑度・可読性・保守性
   - ネストを深くしすぎず、早期 return を使って読みやすくすること。
   - マジックナンバーは用途に応じて `.c` の private 定数か `.h` の共有定数へ寄せること。
   - 名前は役割を表し、呼び出し元から読んだときに意図が分かること。

3. 安全性と堅牢性
   - `malloc` 系の戻り値を確認すること。
   - 失敗経路で `free` / `close` / MLX リソース破棄漏れを起こさないこと。
   - バッファオーバーフロー、未初期化値、NULL 参照などの未定義動作を避けること。

## make check 対象: CR001〜CR015

### CR001: 42 header は不要

ファイル先頭の 42 header はこのリポジトリでは不要。新規追加・生成コードには含めない。

### CR002: include 順序

標準ヘッダ `<...>` を先に置き、その後に自作ヘッダ `"..."` を置く。

### CR003: インデントと空白

インデントはタブを使う。`if` / `for` / `while` と `(` の間に半角スペースを入れる。演算子の前後にも読みやすい空白を入れる。

### CR004: ポインタ表記

`*` は型名側に寄せる。`const` は基本的に型名の前に置く。
ポインタ変数そのものを変更不可にする場合だけ、`* const` の形で書く。

```c
int*	ptr;
const char*	name;
char* const	buffer;
const char* const	label;
```

### CR005: シグネチャとプロトタイプ

戻り値の型と関数名を別行に分ける。関数名と引数の行は先頭にタブを 1 つ入れる。

```c
static int
	parse_value(const char* src, int* out);
```

### CR006: 中括弧と else

関数の `{` は次行、制御構文の `{` は同行末尾に置く。処理が 1 行でも `{}` は省略しない。`else` は `} else {` または `} else if (...) {` の形にする。

### CR007: ファイル末尾

ファイル末尾は改行 1 つで終える。

### CR008: .c のセパレータ

`#include` 群とプロトタイプ宣言またはファイル内定数群の間、全関数定義の直前に次のセパレータを置く。

```c
/* ************************************************************************** */
```

### CR009: .c の配置順序と関数コメント

プロトタイプ宣言は先頭にまとめる。関数定義は公開関数からトップダウン順に置き、プロトタイプ宣言の順序も原則として定義順に合わせる。各関数定義の直前、セパレータの下には `//` で短い日本語コメントを書く。

### CR010: include guard

ヘッダには大文字の include guard を置く。基本形はファイル名由来の `FILENAME_H` とする。名前衝突を避けたい場合は、`UI_FONT_H` のようなディレクトリ prefix 付きも許可する。

```c
#ifndef CONFIG_H
# define CONFIG_H

#endif
```

### CR011: static と公開 API

`.c` 内だけで使う関数は `static` にする。`static` 関数をヘッダへ宣言しない。外部公開する関数は適切なヘッダにプロトタイプを置く。

### CR012: レイヤリング

`common` はモード固有実装へ直接依存しない。FPS/RSP 固有仕様は mode ops や mode 側の関数へ寄せる。

### CR013: 重複定義と名前衝突

マクロやグローバル関数を複数ファイルで重複定義しない。`static` 関数でも、別ファイルで同名が増える場合は意図が分かる名前にする。

### CR014: 未使用関数

どこからも呼ばれない関数を残さない。将来用の dead code は置かず、必要になった時点で追加する。

### CR015: return 文の返却式

値を返す `return` 文では、返却式を必ず括弧で囲む。値を返さない `return ;` は許可する。

```c
return (0);
return (ptr);
return (a + b);
return ;
```

次の形は禁止する。

```c
return 0;
return ptr;
return a + b;
```

## make audit 対象: CR016

### CR016: マジックナンバー

`0` / `1` / `-1` 以外の直書き数値は原則として名前付き定数にする。このルールは `make audit` で助言扱いにする。

次の値はノイズを減らすため検出対象から外す。

- 文字リテラル・文字列リテラル内の数字
- 識別子内の数字（例: `clamp_255`）
- 配列添字・配列サイズの数字（例: `paths[2]`, `buf[256]`）
- 座標中心の `0.5` / `.5`
- 幾何計算で使う `2.0` / `2.` / `/ 2`
- 16進リテラル、ビットシフト幅、RGB などのビット演算値
- 10進文字列変換で使う `9` / `10`

意図的に残す数値は、理由付きで行単位の除外を許可する。

```c
value = 1000; // clint:ignore magic ミリ秒変換
```

理由なしの `clint:ignore magic` は warning とする。

## 人間レビュー対象: CR017〜CR021

### CR017: タブによる整列

変数宣言、構造体メンバ、マクロ定数はタブで縦を揃える。1 行 1 変数を基本とする。同じ型や修飾子の変数は近くにまとめる。

### CR018: ヘッダ内の記述順序

ヘッダ内は、include 群、マクロ定数、構造体・列挙型・typedef、プロトタイプ宣言の順に置く。構造体は `typedef struct s_name { ... } t_name;` 形式を基本とする。

### CR019: ヘッダのセパレータ

ヘッダでも主要なまとまりの間に `.c` と同じセパレータを置く。プロトタイプ 1 個ごとに必ず置くのではなく、関連する公開 API のまとまりごとに置いてよい。

### CR020: ヘッダのコメント

公開マクロ、公開型、公開 API 群には、役割が分かる短い `//` コメントを添える。自明な 1 行プロトタイプすべてにコメントを強制しない。

### CR021: 記述媒体の使い分け（How / What / Why / Why not）

情報は寿命と読者に合った媒体へ書く。

| 媒体 | 書くこと | 例 |
|---|---|---|
| コード | **How** — 名前と構造で意図が読めるようにする | `bfs_to_nearest_patrol()` |
| テスト・検証コード | **What** — 何が保証されるか | 「全 `.cub` でパース結果が改修前後で一致する」 |
| コミットログ | **Why** — なぜこの変更をしたか。設計判断は D-xx や設計書の節番号で参照する | 「衝突判定を move_camera 系へ一本化。2系統の維持は戦闘員統合後に不整合の温床になるため（① §3-C-2）」 |
| 行内コメント | **Why not** — なぜ素直な書き方をしていないか。コードから見えない制約 | 「// 追跡中は視野角ゲートを外す: 背後に回られても一定時間追える仕様のため」 |

- 「次の行が何をするか」を説明する行内コメントは書かない（コードの How と重複するため）。
- **例外**: CR009 が強制する各関数直前の短い日本語コメントと、CR020 のヘッダコメントは**公開 API の What** として従来どおり書く。本ルールは行内コメントとコミットログに適用する。

## lint では完全には判定しないもの

CR017〜CR021、レビュー基準の高凝集・低結合・複雑度・リソース解放の妥当性は、人間のレビューで確認する。必要なら今後 lint を追加する。
