# 戦闘員統合 — 人も NPC も同じリスト

**対象**: チームメンバー全員
**一言でいうと**: **プレイヤーも敵も「戦闘員」という同じ 1 種類にまとめ、
違いは『誰が操作を決めるか』だけ**にした、という話です。
これが「Multiplayer 3+」「Remote players」「AI opponent」の 3 モジュールを
**1 つの仕組みで**成立させる鍵になっています。

---

## 1. 元の cub3D の構造と、その限界

元の cub3D は、はっきり 2 種類に分かれていました。

```
プレイヤー  = カメラ（画面の視点）。世界にただ1人
敵（NPC）   = t_enemy のリスト。複数いる
```

この作りだと、オンライン対戦にするとき困ります。

- 対戦相手（人間）は「敵」なのか「プレイヤー」なのか？
- 4 人対戦なら、残り 3 人はプレイヤーであり、同時に自分から見れば敵でもある
- 席が切断したら、そこを AI に**差し替えたい**。でもプレイヤーと NPC が別構造だと、
  差し替えのたびに**種類そのものを変換**しなければならない

「プレイヤー」と「NPC」を別物として作ると、**接触判定・描画・写真（スナップショット）の
すべてが 2 系統**になり、永久に複雑さを抱えます。

---

## 2. 統合後の考え方

そこで、**全員を「戦闘員（combatant）」という 1 種類のノード**にまとめました。
プレイヤーも、`world.enemies` という**同じリストの中の 1 要素**になります。

```
world.enemies（戦闘員リスト）
  ├─ 戦闘員0  ← 自分（人間）
  ├─ 戦闘員1  ← リモートの人間
  ├─ 戦闘員2  ← AI
  └─ 戦闘員3  ← AI
```

**違いはたった 1 つ、「入力源（input_source）」だけ**です。

```c
// codes/includes/enemy/enemy_types.h
typedef enum e_input_source
{
	INPUT_SRC_AI = 0,        // AI が操作を決める
	INPUT_SRC_EXTERNAL       // 外部（キーボード、将来は WS 経由の人間）が操作を供給
}	t_input_source;
```

- **AI 席**: その場でモード別 AI が「前へ進む」「じゃんけんはグー」などを決める
- **EXTERNAL 席**: 外から届いた入力（`t_input`）をそのまま使う

**移動も当たり判定も描画も、全員まったく同じコードを通ります。**
入力を「AI が作る」か「外から来る」かだけが分岐点です。

---

## 3. これが 3 モジュールを一度に解決する

この 1 つの構造が、課題の 3 モジュールを同じ仕組みで満たします。

| モジュール | 統合構造でどう成立するか |
|---|---|
| **Multiplayer 3+**（4 人対戦） | 戦闘員リストに人間の EXTERNAL 席を 4 つ並べるだけ |
| **Remote players**（リモート対戦） | EXTERNAL 席の入力を「WS 経由の人間入力」にするだけ |
| **AI opponent**（AI 対戦相手） | 席の入力源を AI にするだけ |

そして**切断・再接続**が、この設計の一番おいしいところです。

```
人間がプレイ中          →  input_source = EXTERNAL
       │ 切断
その席を AI が代替       →  input_source = AI      （席はそのまま。中身は動き続ける）
       │ 再接続
人間が戻る               →  input_source = EXTERNAL
```

**席の種類を変換する必要がありません。** `input_source` という値を
`EXTERNAL ⇄ AI` で切り替えるだけ。だから W-12（切断・AI 代替）は、
専用の API 1 つ（`game_set_input_source`）で済みます。

> **元の RSP が、すでにこの形に近かった**のが大きな資産でした。
> RSP は元から「プレイヤー 1 人 + NPC 3 体 = 4 戦闘員」で、じゃんけんの勝敗判定も
> 「全部の異チームペアを見る」作りだったので、統合の差分が最小で済みました。

---

## 4. 「戦闘員」が持つ主な情報

1 つのノード（`t_enemy`）が、席として必要なものを全部持ちます。抜粋すると:

| フィールド | 意味 |
|---|---|
| `input_source` | AI か EXTERNAL か（**唯一の本質的な違い**） |
| `input` | その席が保持している操作（EXTERNAL 席はここに外部入力が入る） |
| `is_player` | 自分の席か（自分の身体は描かない＝カメラになる） |
| `combatant_id` | 席の番号。スナップショット・API が席を特定する鍵 |
| `is_hazard` | マップ由来の敵 'M' か（§5 参照） |
| `death_timer` / `spawn` | 死亡演出と復帰地点（**席ごとに**持つ） |
| `rsp` | RSP の状態（チーム・手・生死） |

`is_player` の席だけは、身体のスプライトを描画リストに繋ぎません。
**自分はカメラ（視点）になるので、自分の身体は画面に映らない**——元の cub3D と同じです。

---

## 5. 例外：ハザード（マップの敵 'M'）は「席」ではない

1 つだけ、戦闘員リストに居るが**席ではない**ものがあります。
FPS モードのマップに置かれる敵 'M'（`is_hazard = 1`）です。

ハザードは「プレイヤーを邪魔する障害物」で、次の点で席と違います。

- アイテムを拾わない
- ゴールに入っても試合を終わらせない
- **接触した席を死なせる側**になる（G-08。当たると数秒リスポーン待ちのペナルティ）

同じリストに置きつつ `is_hazard` で見分けることで、描画（スプライトとして描く）は
共通化しつつ、ルール上の扱いだけ分けています。

---

## 6. スナップショットとの関係

[スナップショットと補間](./スナップショットと補間.md) で「各戦闘員の位置・向き・手を送る」
と書きましたが、**その『戦闘員』がまさにこの統合ノード**です。

送る側は、リストを頭から舐めて 1 ノードずつ写真に写すだけ。
「これはプレイヤー」「これは NPC」という分岐が要りません。
**受け取るブラウザも区別せず、全員を同じスプライト描画で描きます**
（自分の席だけカメラにする）。

統合していなかったら、写真も「プレイヤー欄」と「敵欄」の 2 系統になり、
補間も受け口も倍の複雑さになっていました。

---

## 7. まとめ

1. プレイヤーも NPC も **「戦闘員」1 種類**にまとめ、`world.enemies` の同じリストに置く。
2. 違いは **`input_source`（AI か EXTERNAL か）だけ**。移動・判定・描画は全員共通。
3. これで **Multiplayer 3+ / Remote players / AI opponent が 1 つの仕組み**で成立する。
4. **切断→AI 代替→復帰は `input_source` の切り替えだけ**。席の変換が要らない（W-12）。
5. 元の RSP が既にこの形に近く、**統合の差分が最小**だった。
6. 例外はハザード（'M'）。同じリストに居るが席ではなく、`is_hazard` で見分ける。
7. スナップショットは戦闘員リストをそのまま写すだけ。**分岐が消える**のが最大の利点。
