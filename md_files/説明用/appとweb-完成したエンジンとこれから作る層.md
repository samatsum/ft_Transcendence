# app と web ― 完成したエンジンと、これから作る製品層

**対象**: チームメンバー全員（とくに samatsum / W-10 と、これから app/ を触る人）
**一言でいうと**: **`codes/` + `web/`（C エンジンと WASM デモ）は完成**していて、
**`app/`（TypeScript のオンライン対戦の層）はこれから作る**、という話です。
これまでの説明資料がエンジン中心だったのは、そこが「動いていて説明できるもの」だったからです。

---

## 1. プロジェクトは「2 つの半分」でできている

| 半分 | 実体 | 状態 | 役割 |
|---|---|---|---|
| **エンジン** | `codes/`（C）+ `web/`（WASM とデモ） | ✅ **完成**（Engine E-01〜E-14 / Gameplay G-01〜G-10） | 移動・衝突・敵AI・じゃんけん・描画。**単体で動く** |
| **製品層** | `app/`（backend / frontend / shared） | ⬜ **ほぼ未着手**（W-01 の骨格のみ） | 認証・ロビー・オンライン対戦・履歴。**エンジンをネット越しの製品にする** |

これまでの資料（[pf](./pfプレフィックス関数について.md) / [ループ反転](./ループ反転-誰がゲームを回しているか.md) /
[スナップショット](./スナップショットと補間.md) / [engine_demo](../../web/README.md) …）は
全部**左半分（エンジン）**の話でした。`app/` が出てこなかったのは、**まだ中身が無いから**です。

---

## 2. `app/` の「今」— 骨格だけ

`app/` は W-01 で**枠だけ**作った状態です。ゲームのコードはまだ 1 行も入っていません。

| フォルダ | 今あるもの | 行数感 |
|---|---|---|
| `app/backend/` | Fastify が起動し `/api/health` を返すだけ | 約 40 行 |
| `app/frontend/` | React が `/api/health` を叩いて結果を表示するだけ | 1 画面 |
| `app/shared/` | FE/BE 共有の zod 契約（エラーエンベロープ・health）の骨格 | 数ファイル |

実際、backend の中身の主役はこれだけです。

```ts
// app/backend/src/index.ts（コメントより）
// W-01: Fastify の起動骨格。ここに W-02 で zod 検証、W-03 で Prisma、
// W-08〜W-12 で WS と GameRoom（sim.wasm）が載る。現時点は「起動して疎通する」だけ。
app.get('/api/health', async () => healthSchema.parse({ status: 'ok', ... }));
```

frontend も「backend と疎通できたか」を表示するだけの確認画面で、
**ロビー・GameView・プロフィールは F-01 以降で置き換え**ます。

> つまり `app/` は「**これから作る場所の枠**」。だから説明する中身がまだありません。
> 唯一いま意味があるのは `app/shared` の zod 契約で、「FE/BE が同じ型定義を共有して
> 起動する」ことを実証しています（③§1-B の「フロント/バック双方での検証」の土台）。

---

## 3. `app/` が完成すると、エンジンと「こう繋がる」

`app/` は無関係な別物ではなく、**これまで話した部品（`sim.wasm` / `render.wasm` /
`snapshot_interp.js`）をオンライン対戦として繋ぐ入れ物**です。

| app/ の部分 | これから作る中身 | エンジンとの接続 |
|---|---|---|
| `app/backend/` | REST・認証・DB・**WS と GameRoom** | **`sim.wasm` を Node で回す**（W-10。★samatsum の本丸） |
| `app/frontend/` | ロビー・**GameView**・HUD・プロフィール | **`render.wasm` + `snapshot_interp.js`** を使う（F-06） |
| `app/shared/` | REST/WS のメッセージ契約（zod） | 型の単一情報源。両者が import |

**[④ データの流れ（1 往復）](./スナップショットと補間.md)** を、`app/` で本番実装したものが
オンライン対戦です。`app/backend` が input を受けて `sim.wasm` を回し snapshot を配信、
`app/frontend` が受けて `render.wasm` で描く。**engine_demo（単体）の代わりに、
この 2 つが役割を分担**します。

---

## 4. これから「どこを触る」のか

| 領域 | 今後 |
|---|---|
| `codes/`（C エンジン） | 基本は凍結。例外は FPS のオンライン化（W-14 で `world_delta` 追加）等の小改修のみ |
| `web/` の手書き（engine_demo / sim_demo / bench） | 基本**完成**。開発時のエンジン確認用・**参照実装**として残す（§5） |
| `web/build/`（`*.wasm`） | codes/ を直すたび `make web` / `make sim` が**作り直す**（生成物） |
| **`app/`** | **これから本格実装**。W-02〜W-16 / F-01〜F-12。**当面の作業の中心はここ** |

**次に手を動かすと、話の中心は一気に `app/` へ移ります。** とくに `app/backend` の
**W-10（`sim.wasm` を回して WS 配信）が samatsum の次の本丸**です。

---

## 5. 補足：`web/` は今後も触る？

「ほぼ完成だが、3 つの経路で今後も関わる」が答えです。

1. **`codes/` を直すと `web/build/*.wasm` が再生成される**。手で `web/` を編集するのでは
   なく、C を直して `make web` / `make sim` が生成物を更新する（例: W-14 の `world_delta`）。
2. **`snapshot_interp.js` は F-06 が流用する**。`app/frontend` から import / 取り込む形になり、
   実質 `web/` の資産が `app/` に持ち込まれる（補間ロジックを書き直さずに済む）。
3. **wasm の置き場や読み込み経路の見直し**。`app/` とエンジンを結線する W-10 / F-06 の際、
   `web/build/` の成果物をどこから読むかを再検討する可能性がある（設計上の未決事項）。

`engine_demo` / `sim_demo` / `bench_render.mjs` そのものを積極的に書き換える予定は薄く、
**「動く参照実装」として残す**のが基本方針です（F-06 は `replay.html` を下敷きにする）。

---

## 6. まとめ

1. プロジェクトは **エンジン（`codes/`+`web/`、完成）** と **製品層（`app/`、これから）** の 2 つ。
2. 説明資料がエンジン中心だったのは、そこが**動いていて説明できるもの**だったから。
3. `app/` は今は **W-01 の骨格だけ**（`/api/health` と zod 契約）。ゲームのコードは未実装。
4. 完成すると `app/` が **エンジンと繋がる層**になる：backend が `sim.wasm`、frontend が
   `render.wasm` + `snapshot_interp.js` を使う。
5. **これからの作業の中心は `app/`**。とくに `app/backend` の **W-10 が samatsum の本丸**。
6. `web/` は基本完成だが、C の再ビルド・`snapshot_interp` 流用・置き場見直しで今後も関わる。
