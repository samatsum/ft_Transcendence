# `.cub` マップの読み方

**対象**: チームメンバー全員（とくに samatsum / W-14 と、マップを触る人）
**一言でいうと**: **ゲームのステージを 1 枚のテキストファイルで表したもの**が `.cub` です。
上半分が「設定」、下半分が「地図（文字で描いた見取り図）」という構成です。

> このドキュメントは**全体像をつかむ**ためのものです。1 文字ずつの厳密な仕様は
> [ユーザードキュメント §4](../04_エンジン資料/ユーザードキュメント.md)（`.cub` 記述ルール）が
> 正本なので、実際にマップを書く・直すときはそちらを参照してください。

---

## 1. `.cub` は 2 段構成

```
┌─ 設定ヘッダ ────────────────┐
│ R 1920 1080                 │  解像度
│ NO ./textures/wall/...xpm   │  壁テクスチャ（北）
│ F 184,113,39                │  床の色
│ ...                         │
├─ マップ本体 ────────────────┤
│ 1 1 1 1 1 1 1               │  1 = 壁
│ 1 0 0 0 N 0 1               │  0 = 床、N = スポーン地点
│ 1 1 1 1 1 1 1               │
└─────────────────────────────┘
```

- **上（設定ヘッダ）**: 解像度・テクスチャのファイル名・色・敵の速度などを
  `キー 値` の形で並べる。
- **下（マップ本体）**: `1`（壁）や `0`（床）などの文字を並べて、**上から見た
  ステージの見取り図**を描く。

---

## 2. マップ本体で使う主な文字

`.cub` のマップ部分は、決まった文字だけで構成されます
（C 側の許可集合 `VALID_MAP_CHARACTERS`＝`config.h`）。よく使うものだけ抜き出すと:

| 文字 | 意味 |
|---|---|
| `1` | 壁（通れない） |
| `0` | 床（通れる） |
| `N` `S` `E` `W` | **スポーン地点**（戦闘員の初期位置。向きも表す） |
| `M` | 敵（ハザード）。巡回し、見つけると追跡してくる |
| `P` | 敵の巡回路 |
| `k`〜`o` | 収集アイテム（星） |
| `D` | 扉（**全部集めるまでは壁**、集め終わると開く） |
| `G` | **ゴール**（先に踏んだ戦闘員が勝ち。FPS のレース用） |
| `a`〜`e` / `f`〜`j` | 障害物オブジェクト（通行不可 / 通行可） |

> **重要**: マップ文字 `G`・`D`・`M` は**新設したものではなく、元の cub3D に既にありました**。
> オンライン対戦化（G-06〜G-08）でやったのは「勝者を席に帰属させる」等のルール調整で、
> マップ書式そのものは足していません。

---

## 3. RSP と FPS の区別は「置き場所」で決まる

同じ `.cub` でも、**どのフォルダに置くか**でモードが決まります。

```
maps/rsp_map/  に置いたマップ  →  RSP モード（2vs2 じゃんけん）
maps/fps_map/  に置いたマップ  →  FPS モード（1vs1 レース）
```

これは native も web も同じルールです。`main.c`（native）も `web_init` の呼び出し側（JS）も、
**パスに `rsp_map/` が含まれるか**を見てモードを切り替えます。ファイルの中身に
「これは RSP です」とは書きません。**置き場所がモードの宣言**です。

### スポーン文字とチームの対応（RSP）

RSP では、スポーン文字がそのまま**チーム分け**になります。

| 文字 | チーム |
|---|---|
| `N` `W` | 赤 |
| `S` `E` | 青 |

だから RSP マップは、赤 2 席・青 2 席が別マスに立つように `N` `W` `S` `E` を
配置します（先ほどの `rsp.cub` は各文字が 2 つずつありました）。

---

## 4. FPS の「関門レース」の作り方

FPS のオンライン対戦マップ（例 `fps_duel.cub`）は、`D` と `G` を組み合わせて
**関門付きのレース**を作ります。

```
収集アイテム(k〜o)を全部集める → 扉(D)が開く → ゴール(G)へ走る → 先に踏んだ方が勝ち
```

- `G` を**置かない**マップは、「全部集めて扉を開ける」ところまでが目的（従来の cub3D 的な遊び）。
- `M`（敵ハザード）を通り道に置くと、接触＝数秒リスポーン待ちの**妨害**になります（G-08）。

この「収集 → 扉 → ゴール」の関門が最後まで通れることは、
`make test` の G-09 検査が**自動で確認**しています。

---

## 5. オンライン対戦では「ホワイトリスト」しか使わない

`maps/` には開発・テスト用の `.cub` がたくさんありますが（`door_test.cub` など）、
**オンライン対戦で使えるのは、サーバに登録された 4 枚だけ**です。

| id | 名前 | モード | ファイル |
|---|---|---|---|
| `rsp` | （RSP 標準） | rsp | `maps/rsp_map/rsp.cub` |
| `rsp_pillars` | Pillars | rsp | `maps/rsp_map/rsp_pillars.cub` |
| `21x21_arena` | Arena 21 | fps | `maps/fps_map/21x21_arena.cub` |
| `fps_duel` | Duel Run | fps | `maps/fps_map/fps_duel.cub` |

これを**ホワイトリスト**と呼びます（③§2-E）。理由は 2 つ。

1. **安全**: 利用者が任意のファイルパスを送ってサーバのファイルを読ませる、といった
   攻撃を防ぐ。サーバは「登録済みの id」しか受け付けない。
2. **品質**: 対戦バランスをテストプレイで確認したマップだけを出す。

### 流れ（W-14 の仕事）

```
クライアント「マップ id = fps_duel で対戦したい」
        │
サーバがホワイトリスト表で id → ファイルパスを解決
        │
サーバが .cub を読み、その中身（テキスト）を welcome.map_text として配布
        │
各ブラウザは受け取ったテキストで render.wasm を初期化して描く
```

ポイントは、**サーバがマップの中身そのものを配る**ことです。クライアントに
ファイルパスを渡してファイルを読ませたりはしません（②§5-B）。

---

## 6. まとめ

1. `.cub` は **上＝設定ヘッダ / 下＝文字で描いた地図** の 2 段構成。
2. マップは決まった文字だけで描く（`1`壁 `0`床 `N/S/E/W`スポーン `M`敵 `D`扉 `G`ゴール…）。
3. `G` `D` `M` は**元からある文字**。オンライン化で新設していない。
4. **RSP か FPS かは、置くフォルダ（`rsp_map/` / `fps_map/`）で決まる**。
5. RSP はスポーン文字がチーム分け（`N`/`W`=赤、`S`/`E`=青）。
6. オンラインでは**ホワイトリスト登録の 4 枚だけ**。サーバが中身を配る（W-14）。
