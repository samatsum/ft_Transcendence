# 3 つのビルドターゲット — make / make web / make sim

**対象**: チームメンバー全員
**一言でいうと**: **同じ C のソースから、用途の違う 3 つの成果物を作り分けている**、
という話です。どのコマンドが何を作り、何のために使うのかを 1 枚にまとめます。

前提として [pfプレフィックス関数について](./pfプレフィックス関数について.md) と
[ループ反転](./ループ反転-誰がゲームを回しているか.md) を読んでおくと、
「なぜ 3 つに分けられるのか」が腑に落ちます。

---

## 1. まず全体像

```
                共通のゲームロジック（移動・衝突・敵AI・じゃんけん・スコア）
                        codes/srcs/common, fps, rsp
                                    │
        ┌───────────────────────────┼───────────────────────────┐
        │                           │                           │
    make                        make web                    make sim
        │                           │                           │
   platform/native            platform/web               platform/headless
   ＋描画すべて                ＋描画すべて                 描画は除外
        │                           │                           │
        ▼                           ▼                           ▼
   cub3D（ネイティブ）        render.wasm（ブラウザ）      sim.wasm（サーバ）
```

**ゲームロジックのソースは 3 つとも同じ**。違うのは
「どの [`pf_*`](./pfプレフィックス関数について.md) 実装を混ぜるか」と
「描画のコードを入れるか外すか」だけです。

---

## 2. 4 つのコマンド

| コマンド | 成果物 | 何のためのものか |
|---|---|---|
| `make` | `cub3D`（Linux 実行ファイル） | **開発・回帰確認**用。従来の cub3D そのもの |
| `make web` | `web/build/render.js` + `.wasm` | **ブラウザで描く**ためのもの。画面担当 |
| `make sim` | `web/build/sim.js` + `.wasm` | **サーバで判定する**ためのもの。頭脳担当 |
| `make test` | `codes/tests/sim_test`（実行ファイル） | **受入テスト**。85 検査を自動実行 |

（`make test` は成果物というより検証用ですが、ビルドの一種なので並べています。）

---

## 3. それぞれ詳しく

### make（native）— 開発の足場

```bash
make          # → ./cub3D が出来る
./cub3D maps/rsp_map/rsp.cub
```

- MiniLibX を使い、X11 のウィンドウを開いて動く**従来どおりの cub3D**。
- **役割**: 全期間を通じて「壊れていないこと」を確認する基準。
  エンジンをいじったら、まず `make` して両モードが動くかを見ます。
- 3 つのうち**唯一 MiniLibX を必要とする**ビルドです
  （[pf の資料](./pfプレフィックス関数について.md) の native 実装）。

### make web（render.wasm）— 画面担当

```bash
make web      # → web/build/render.js + render.wasm
```

- ブラウザの `<canvas>` に描くための WASM。
- **役割**: サーバから届いたスナップショットを**描く**だけ。
  **自分ではゲームを判定しません**（[スナップショットと補間](./スナップショットと補間.md)）。
- `make web` は最初に `web-assets`（XPM 画像を RGBA へ変換）も走らせます。
  ブラウザは XPM を読めないので、事前変換が要ります。
- 動かすには HTTP サーバが要ります（`file://` では wasm を fetch できない）。

### make sim（sim.wasm）— 頭脳担当

```bash
make sim      # → web/build/sim.js + sim.wasm
```

- Node サーバの中で回す、**画面を持たない**ゲーム本体。
- **役割**: 唯一の正しい状態を計算する（サーバ権威）。
  30Hz で `game_step` を回し、状態の写真（スナップショット）を返す。
- ここが肝心な点: **sim.wasm には描画のコードが物理的に入っていません**。
  Makefile の `SIM_RENDER_EXCLUDES` が `draw.c` / `raycast.c` / `sprite.c` /
  `font.c` などをリンク対象から外しています。
  だから「描画シンボルが無い」状態で成立します（[pf の資料](./pfプレフィックス関数について.md) の headless）。

### make test（受入テスト）

```bash
make test     # → sim_test をビルドして即実行。85 検査
```

- `sim.wasm` と**同じソース構成**（headless）を、emcc ではなく**普通の gcc**で
  ネイティブの実行ファイルにしたもの。ブラウザも Node も要らずに走ります。
- **役割**: G-05〜G-09 のルール（先取点・ゴール・スポーン・ハザード・マップ）が
  仕様どおり動くかを自動検証。**エンジンを壊す変更を検知する安全網**。

---

## 4. 「描く担当」と「判定する担当」に分けた理由

`render.wasm`（画面）と `sim.wasm`（頭脳）が別なのは、
[スナップショットと補間](./スナップショットと補間.md) のサーバ権威モデルの直接の帰結です。

| | render.wasm | sim.wasm |
|---|---|---|
| どこで動く | ブラウザ | Node サーバ |
| 描画する | ✅ する | ❌ しない |
| ゲームを判定する | ❌ しない | ✅ する（唯一の正） |
| 何個動く | 各プレイヤーに 1 つ | 1 試合に 1 つ（複数試合なら複数） |

**判定を 1 か所（sim.wasm）に集めるからチートもズレも消える**。
**描画を各ブラウザ（render.wasm）に任せるから軽い**。役割を割ったわけです。

---

## 5. CI が 3 つとも毎回ビルドする

GitHub Actions（[.github/workflows/ci.yml](../../.github/workflows/ci.yml)）が、
push のたびに **3 ターゲット全部**をビルドし、`make check`（lint）・`make test`・
native の起動スモークまで走らせます。

これは単なる確認ではなく**安全装置**です。たとえば `common/` に MLX を持ち込むと
`make web` / `make sim` が落ちるので、**共通コードを壊す変更がすり抜けられません**
（[pf の資料 §5](./pfプレフィックス関数について.md)）。

「3 つとも緑になった」＝「3 環境どれも壊していない」の保証になります。

---

## 6. まとめ

1. **同じゲームロジックから、用途の違う 3 つを作り分けている**。
2. `make`＝開発の足場（従来 cub3D、MLX 使用）／ `make web`＝**描く**／
   `make sim`＝**判定する**／ `make test`＝受入テスト。
3. `sim.wasm` は**描画コードごとリンクから外して**いる（頭脳だけ）。
4. 描画（render）と判定（sim）を分けたのは、**サーバ権威モデルの帰結**。
5. **CI が 3 つとも毎回ビルド**し、共通コードを壊す変更を止める。
