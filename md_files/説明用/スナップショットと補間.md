# スナップショットと補間 — サーバだけが正しい世界

**対象**: チームメンバー全員
**一言でいうと**: **本当のゲームはサーバの中だけで進み、ブラウザはその「写真」を
受け取って描くだけ**にした、という話です。ブラウザは勝敗を一切決めません。

前提として [ループ反転](./ループ反転-誰がゲームを回しているか.md) を読んでおくと理解が早いです。

---

## 1. なぜ「サーバだけが正しい」必要があるのか

4 人でオンライン対戦をするとき、素朴に作るとこうなります。

```
各プレイヤーのブラウザが、それぞれ自分でゲームを計算する
```

これは**必ず破綻します**。理由は 2 つ。

1. **チート**: ブラウザのコードは利用者が書き換えられます。「自分は無敵」「相手に
   勝った」と自分の PC が言い張れてしまう。**評価で必ず突かれる**弱点です。
2. **ズレ**: 通信には遅延があり、各自の計算が少しずつズレます。「A の画面では勝ち、
   B の画面では負け」が起きて、どちらが本当か決められません。

そこで、**判定する場所を 1 つに決めます**。それがサーバです。

> **サーバ権威（server-authoritative）**: ゲームの本当の状態は**サーバの中だけ**に
> 存在し、勝敗・スコア・位置はサーバが決める。ブラウザはそれを**表示するだけ**。
> この方式にすると、チートは「サーバが認めないので無効」、ズレは「全員が同じ
> サーバの状態を見るので起きない」と、両方が原理的に消えます。

**最重要の帰結**: **ブラウザ側（クライアント）に勝敗を決めるコードは 1 行も無い。**
これは設計の建前ではなく、実際に `game_apply_snapshot` のコメントにも
「勝敗・スコアの正本はサーバであり、ここでは一切判定しない」と明記して守っています。

---

## 2. スナップショット＝ゲームの「写真」

サーバは 1 秒間に 30 回（30Hz）ゲームを進めます（`game_step`）。
その途中の状態を、定期的に**写真**に撮ってブラウザへ送ります。これがスナップショットです。

写真に写っているのは、**表示に必要な最小限**だけです。

| 中身 | 例 |
|---|---|
| 試合の状態 | playing / finished、勝者、スコア |
| 各戦闘員 | id・チーム・手・**位置(x,y)**・**向き**・生存・AIか・復帰までの秒 |

1 回分で **1KB 未満**に収まります（4 人 + 数体なので JSON で十分。実測で確認済み）。
正本のレイアウトは [codes/includes/platform/sim.h](../../codes/includes/platform/sim.h) です。

### 誰が何を担当するか（ここが W-10 と F-06 の境界）

```
サーバ（Node）                            ブラウザ
─────────────                            ─────────
game_snapshot()                          受信
  → フラットな数値の配列 ──JSON化──▶ WS ──▶ JSON
                                            → 補間（§3）
                                            → game_apply_snapshot()
                                            → 画面に描画
```

- **数値を JSON の文字列にするのは Node の仕事**（sim.wasm はしない）。
- **tick 番号（何枚目の写真か）を振るのも Node**。sim は番号を持ちません。

この 2 点は「サーバ側の責務」として設計で切ってあります（②§5-C）。

## 3. 補間（interpolation）＝ 写真と写真の「あいだ」を作る

ここで問題が起きます。

- サーバは 1 秒に 30 回進むが、配信はもっと粗い（例: 15〜20 回/秒）
- ブラウザは 1 秒に 60 回描く

つまり **写真の枚数より、描く回数のほうが多い**。写真をそのまま出すと、
**キャラクターがカクカク**（写真が届いた瞬間だけワープする）動きます。

そこで **2 枚の写真の「あいだ」を計算で埋めます**。これが補間です。

```
写真A（0.0秒: 位置10）        写真B（0.1秒: 位置20）
        │◀────── この間を描くとき ──────▶│
        位置10  12  14  16  18  20  と滑らかに埋める
```

### 補間のしくみ（数値例つき）

名前は難しそうですが、**やっているのは割り算とかけ算だけ**です。式は 1 本。

```
描く値 = A + (B - A) × alpha
```

`alpha`（アルファ）は **「A と B の、どのへんか」を 0〜1 で表した割合**です。
alpha=0 ならちょうど A、0.5 ならまんなか、1 ならちょうど B。

具体例。写真A（時刻 0.00 秒・位置 10）と 写真B（時刻 0.10 秒・位置 20）があり、
いま **0.07 秒**の絵を描きたいとします。

**手順1 — alpha（どのへんか）を出す**

```
alpha = (描きたい時刻 - Aの時刻) / (Bの時刻 - Aの時刻)
      = (0.07 - 0.00)  / (0.10 - 0.00)
      = 0.7                          ← A と B の 70% の地点
```

**手順2 — その割合で混ぜる**

```
位置 = 10 + (20 - 10) × 0.7 = 17     ← 位置 17 に描く
```

これを**毎フレーム（1 秒 60 回）**やります。時間が進むと alpha が 0 → 0.2 → 0.5 …と
増えるので、キャラは A の位置から B の位置へ**すーっと滑って**いきます。
アニメの「中割り」と同じ発想です。

```
0.00秒 alpha=0.0 → 位置10
0.05秒 alpha=0.5 → 位置15
0.07秒 alpha=0.7 → 位置17
0.10秒 alpha=1.0 → 位置20   ← ここで次の写真Cに切り替え、また alpha=0 から
```

実際のコード（[web/sim_demo/replay.js](../../web/sim_demo/replay.js)）も、この 2 手順そのものです。

```js
alpha = (playT - timeOf(cur)) / (timeOf(next) - timeOf(cur));  // 手順1
alpha = Math.min(1, Math.max(0, alpha));                       // 0〜1 に収める
const flat = window.Cub3DSnapshotInterp.interpolate(cur, next, alpha);  // 手順2
```

> **補間は「予測」ではありません。** A も B も**すでに手元にある確定データ**です。
> やっているのは「既知の 2 点の**あいだ**を埋める」ことで、未来を当ててはいません。
> だから外れようがない。これが「100ms 過去を描く」理由（後述）とつながります。

### 「同じ物体」はどう対応づけるのか — id で照合

「A の物体 a と B の物体 a を、どうやって同じものだと判定しているのか」。
答えは、**戦闘員が持つ変わらない id（席番号）で突き合わせている**、です。

1. 写真B を「id → 戦闘員」の**早見表（Map）**にする
2. 写真A の戦闘員を 1 人ずつ見て、**同じ id を B の早見表から引く**
3. 見つかった 2 つ（A のその席・B のその席）の位置・向きを補間する

```js
const later = new Map(d1.combatants.map((c) => [c.id, c]));  // B を id で引ける表に
d0.combatants.forEach((c) => {
	const next = later.get(c.id);   // A の戦闘員 c と同じ id を B から探す
	if (!next) return;              // B に居なければ補間しない（A の値のまま）
	// c（A側）と next（B側）の位置・向きを補間する
});
```

**リストの並び順では照合していません**（並びが違っても平気）。
id という「背番号」で突き合わせるので、途中で順番が変わっても・誰かが消えても
正しく対応します（消えた席は受け口 `game_apply_snapshot` が画面外へ退避させます）。

### ピクセルは混ぜていない。状態を混ぜて描き直す

ここが一番大事で、直感と違うところです。
**2 枚の写真のピクセル（色）は、一切混ぜていません。**

```
写真A・写真B（各戦闘員の位置・向き）
      │  id で照合して、位置と向きだけ補間
      ▼
「いまの瞬間の全戦闘員の位置・向き」という数値
      │  これを C の t_game に書き込む
      ▼
C がその位置から 3D シーンを丸ごと描き直す（レイキャスト）
      ▼
新品のピクセルが毎フレーム出来上がる
```

つまり **「絵と絵を混ぜている」のではなく、「登場人物を正しい位置に置き直して、
毎フレーム 3D を一から描いている」**。もしピクセルを混ぜたら、半透明の残像のような
気持ち悪い絵になります。実際は「敵は座標(17,8)・向き 0.7」と**数値を確定してから、
その敵を通常どおりくっきり描く**ので、いつも鮮明です。

**補間の対象は「数十個の数値」だけ**です（4 人 × 位置 2 + 向き 1 ＝ 12 個ほど）。
画面は約 96 万ピクセルありますが、そのどれも補間していません。
**だから軽い**——ここが直感に反する肝です。

### 実物（[web/snapshot_interp.js](../../web/snapshot_interp.js)）

```js
// 位置は線形（まっすぐ按分）
out[o + 3] = c.pos[0] + (next.pos[0] - c.pos[0]) * alpha;   // x
out[o + 4] = c.pos[1] + (next.pos[1] - c.pos[1]) * alpha;   // y
// 向きは「最短の回り方」で
out[o + 5] = lerpAngle(c.dir, next.dir, alpha);
```

`alpha` は「A と B のどのあたりか」を 0〜1 で表す割合です。

**向きだけ特別扱い**しているのに注目してください。角度は 360°＝0° で一周するので、
359° から 1° へ動くとき、素直に按分すると**逆回り（長いほう）に 358° ぐるっと回って
見える**。それを防ぐため `lerpAngle` が「短いほうの弧」で回します。

```js
function lerpAngle(a, b, alpha) {
	let diff = b - a;
	while (diff > Math.PI)  diff -= 2 * Math.PI;   // 半周を超えたら
	while (diff < -Math.PI) diff += 2 * Math.PI;   // 逆側から回る
	return a + diff * alpha;
}
```

### 位置と向きは補間する。でも「勝敗」は補間しない

補間するのは **位置・向き**という連続的に動く値だけです。
**スコア・生死・勝者・手（グー/チョキ/パー）は、古いほうの写真の値をそのまま使います**。

これらは「7 点」「死んでいる」のように飛び飛びの値で、**あいだを作る意味が無い**し、
勝手に作ると**サーバの判定と食い違う**からです。ここでも「クライアントは判定しない」が
徹底されています。

### なぜ「100ms 前」を描くのか

補間には「A と B の 2 枚」が必要なので、**B が届くまで A のあとを描けません**。
そこで、ブラウザは**わざと 100ms 過去の時刻を描きます**。こうすると、
描こうとする時刻の前後の写真が**必ず手元にある**状態になり、常に補間できます。

代償は「見えている世界が 100ms 遅れる」ことですが、**視点の回転だけは例外**で、
自分のマウス操作は**待たずに即座に反映**します（回転はサーバの許可を待たなくても
ズレないため）。これで操作の手応えは保たれます。

## 4. ブラウザ側の受け口 — 表示専用モード

補間した数値を、C の描画エンジンに渡すのが `game_apply_snapshot` です。

```c
// codes/srcs/platform/web/web_snapshot.c
int game_apply_snapshot(t_game* game, const double* snap, int len, int view_id)
{
	// snap（補間済みの写真）を、表示用の t_game に書き込むだけ。
	// 勝敗・スコアの正本はサーバであり、ここでは一切判定しない。
	// view_id の席＝自分はカメラになり、他の席は敵スプライトとして描く。
}
```

ポイントは、**このモードでは `game_step` を呼ばない**ことです。
ローカルで計算するとサーバの正本とケンカするので、ブラウザは
「写真を貼り替えて描くだけ」に徹します。
（[ループ反転](./ループ反転-誰がゲームを回しているか.md) の `web_render` ではなく、
`web_render_frame`＝描画だけの関数を使う、というのがこの違いです。）

`view_id` は「4 人のうち自分がどの席か」です。自分の席はカメラ（一人称視点）になり、
残り 3 席は敵スプライトとして描かれます。

## 5. これが動く実例

本番の Node サーバ（W-10）はまだ未実装ですが、**一方通行の全経路が動くデモ**があります。

```
web/sim_demo/record.mjs    sim.wasm を回して写真を撮り、snapshots.json に保存
        │  （＝サーバ役）
        ▼
web/sim_demo/replay.html   snapshots.json を読み、補間して描画（＝ブラウザ役）
```

**F-06 は、この `replay.html` の「`snapshots.json` を読む」ところを
「WS で受信する」に差し替えるだけ**で GameView になります。
補間（`snapshot_interp.js`）と受け口（`game_apply_snapshot`）はそのまま流用できます。

## 6. まとめ

1. **本当のゲームはサーバの中だけで進む**（サーバ権威）。チートとズレを原理的に消すため。
2. **クライアントに勝敗判定コードは無い**。これは徹底して守られている。
3. サーバは状態の**写真（スナップショット）**を定期配信する。1KB 未満・JSON。
4. 写真は粗いので、**2 枚のあいだを補間**して滑らかに描く。位置と向きだけ。
   スコア・生死は補間せず古い写真の値を使う（判定しないから）。
   補間は `A + (B-A)×alpha` の按分で、**予測ではなく既知 2 点のあいだ埋め**。
   物体の対応は **id で照合**し、**ピクセルではなく数値を混ぜて毎フレーム描き直す**。
5. **100ms 過去を描く**ことで常に補間できる。ただし**自分の視点回転だけは即時**。
6. ブラウザの受け口は `game_apply_snapshot`（描くだけ、`game_step` しない）。
7. `web/sim_demo/` に全経路の動く実例がある。**F-06 はこれを WS に繋ぎ替えるだけ**。
