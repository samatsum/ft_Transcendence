# どの言語がどこで動くのか

**対象**: チームメンバー全員
**一言でいうと**: **「書く言語」と「動く言語」は別物**、という話です。
TypeScript はどこでも実行されず、走る前に JavaScript になります。
ブラウザも Node も、動かせるのは **JavaScript と WASM だけ**です。

前提として [なぜNodeでCを動かせるのか-WASM入門](./なぜNodeでCを動かせるのか-WASM入門.md) を
読んでおくと、この文書の「WASM」がすっと入ります。

---

## 1. 大前提：TypeScript は「実行されない」

意外かもしれませんが、**TypeScript というプログラムが動く場所は、
ブラウザにもサーバにも存在しません。**

TypeScript は**書くための言語**で、動かす前に必ず **JavaScript に変換**されます。
型チェックという安全装置が付いた JavaScript、と思ってください。
**走る前に型は外され、ただの JavaScript になります。**

> 例えるなら、TypeScript は「補助輪付きの JavaScript」。
> **走り出す前に補助輪（型）が外れて、実際に走るのは JavaScript** です。

だから、**ランタイム（ブラウザや Node）に渡せる形**は 3 つだけです。

- **機械語**（native の C から。Linux が直接動かす）
- **JavaScript**（ブラウザと Node が受け取る）
- **WASM**（ブラウザと Node が受け取る）

**ブラウザも Node も、TypeScript は受け取れません。JavaScript と WASM だけ**です。

> **ただし**——「JavaScript や WASM は機械語ではないの?」と思ったら、その勘は正しいです。
> **最終的には全部、機械語になって CPU が動かします**（CPU は機械語しか分からない）。
> 上の 3 つは「CPU に届く直前の形」であって、違うのは**いつ機械語にするか**です。次節へ。

---

## 2. 実は、全部いずれ機械語になる — 違いは「いつ」

**CPU は機械語しか実行できません。例外はありません。**
だから JavaScript も WASM も、最後は必ず機械語になって CPU が走らせます。
「実行される言語が 3 種類ある」のではなく、**機械語にするタイミングと配る形が違う**だけです。

| | いつ機械語になるか | 配る（ship する）もの |
|---|---|---|
| **C（native）** | **ビルド時**（あなたが `gcc` で先に翻訳） | 機械語そのもの（`cub3D`） |
| **JavaScript** | **実行時**（ブラウザ/Node の中で自動的に） | ソースコード（文字のまま） |
| **WASM** | **実行時**（同上。ただし高速・確実） | バイトコード（機械語に近い中間形式） |

- **C** は「先に全部機械語へ翻訳してから配る」＝ **AOT（Ahead-Of-Time、事前コンパイル）**。
- **JavaScript / WASM** は「そのまま配って、動かす瞬間に機械語へ翻訳する」＝
  **JIT（Just-In-Time、実行時コンパイル）**。

ブラウザや Node の中には **V8** という**エンジン**が入っていて、こいつが JavaScript を
読みながら**その場で機械語に変換して走らせて**います。
「JS が動く」の実体は「V8 が JS を機械語にして CPU に流している」ことです。

### では、なぜ WASM のほうが速いのか

「両方とも機械語になるなら速さは同じでは?」——**同じ機械語でも"質"が違います**。

- **JavaScript** は型が実行時まで決まりません（`x` が数値か文字列か、動かすまで不明）。
  だからエンジンは**「たぶん数値だろう」と推測して機械語を作り、外れたら作り直す**という
  保険とやり直しを抱えます。これが遅さの元です。
- **WASM** は C 由来で**型がガチガチに決まっている**ので、**推測なしで素直に速い機械語**を
  吐けます。だから C に近い速度が出ます。

レイキャストのような重い計算を C/WASM に任せる理由がこれです
（[なぜブラウザは並列描画しないのか](./なぜブラウザは並列描画しないのか.md) の「速度が足りていた」の裏付け）。

---

## 3. 「書く言語」と「動く言語」を分ける

ここを分けて考えると、混乱が消えます。

| | 種類 |
|---|---|
| **書く言語**（人間が書く） | C / TypeScript |
| **動く言語**（機械が実行する） | 機械語 / JavaScript / WASM |

書いたものは、実行前に必ず**変換**されます。

- **C** → WASM（`emcc`）または 機械語（`gcc`）
- **TypeScript** → JavaScript（Vite / tsx）

**「書く言語 = 動く言語」ではない**のが、Web 開発の分かりにくさの正体です。

（ここでいう「動く言語」は §2 のとおり **ランタイムに渡す形**の意味で、
機械語 / JavaScript / WASM のいずれも**最終的には CPU 上で機械語として走ります**。）

---

## 4. 言語 → いつ変換 → どこで動く

| 書く言語 | いつ変換される | 何になる | 最終的に動く場所 |
|---|---|---|---|
| **C**（エンジン） | ビルド時（`emcc`） | WASM | **ブラウザ と Node** の両方 |
| C（エンジン・native 用） | ビルド時（`gcc`） | 機械語 | Linux が直接 |
| **TypeScript**（フロント） | ビルド時（Vite） | JavaScript | **ブラウザ** |
| **TypeScript**（バック） | 起動時（tsx） | JavaScript | **Node（サーバ）** |

すべて「実行前に、機械が動かせる形へ変換される」——ここが共通点です。

> **補足**: バックエンドは `tsx` というツールで動かします。tsx は TypeScript を
> **その場で JavaScript に変換してから Node に渡す**ので、体感は「TS をそのまま実行」
> ですが、中身は「TS → JS → Node」です。フロントの Vite も同じで、TS を JS へ変換して
> ブラウザに渡します。**どちらも走るのは JavaScript** です。

---

## 5. 実行時、各場所で動いているもの

ブラウザとサーバ、**どちらも中身は「JavaScript ＋ WASM」の2層**です。

```
┌─ プレイヤーのブラウザ ─────────────┐        ┌─ あなたのサーバ（Node）─────────┐
│                                     │  通信   │                                 │
│  JavaScript  ← フロントの TS から   │  (WS)   │  JavaScript  ← バックの TS から │
│  （画面・入力・通信の"手"）         │◀──────▶│  （通信・DB・タイミングの"手"） │
│      │ 呼ぶ                         │        │      │ 呼ぶ                     │
│      ▼                              │        │      ▼                          │
│  render.wasm ← C エンジンから       │        │  sim.wasm  ← C エンジンから     │
│  （絵を計算する"脳"）               │        │  （勝敗を計算する"脳"）         │
└─────────────────────────────────────┘        └─────────────────────────────────┘
```

前の資料でいう「**脳（WASM）と手（JavaScript）**」のペアが、**両方の場所にいる**イメージです。

| 場所 | JavaScript（手）の担当 | WASM（脳）の担当 |
|---|---|---|
| **ブラウザ** | Canvas に描く・キー入力・サーバと通信 | 画面のピクセルを計算（`render.wasm`） |
| **Node サーバ** | WS 通信・DB・30Hz のタイミング | 移動・勝敗を計算（`sim.wasm`） |

**手（JS）と脳（WASM）が、同じ場所でペアを組んでいる**。これが両方に共通の構造です。

## 6. TypeScript は「3 つ」ではなく「2 か所で書く」

`app/` の中に TypeScript が 3 つありますが、これは**用途と行き先で分かれている**だけです。

| フォルダ | 書く言語 | 変換後 | 動く場所 |
|---|---|---|---|
| `app/frontend/` | TypeScript | JavaScript | **ブラウザ** |
| `app/backend/` | TypeScript | JavaScript | **Node（サーバ）** |
| `app/shared/` | TypeScript | 両方に取り込まれる | ブラウザ と Node の両方 |

**同じ TypeScript でも、フロント用は最終的にブラウザ、バック用は Node** で動きます。
`app/shared/`（zod の型）は、フロントとバックの**両方に取り込まれて**、それぞれの
JavaScript の一部になります。書く言語は同じでも、**変換後に置かれる場所が違う**わけです。

## 7. よくある勘違い

> **Q: TypeScript はブラウザで動くの？**
> A: 動きません。ビルドで JavaScript に変換され、**ブラウザが動かすのは JavaScript** です。
> 「TypeScript が動いている」ように見えても、実体は変換後の JavaScript です。

> **Q: JavaScript と WASM は、別々に動く別のプログラム？**
> A: いいえ、**ペアで動きます**。JavaScript が主役（ホスト）で、WASM を読み込んで
> その関数を呼びます。WASM は計算専門で、画面や通信には JavaScript を通してしか触れません。

> **Q: ブラウザの JavaScript と Node の JavaScript は同じもの？**
> A: 言語としては同じ JavaScript です。違うのは**居場所と権限**。ブラウザの JS は
> セキュリティの檻の中（ファイルは触れない）、Node の JS は檻の外（ファイルも通信もできる）。
> 詳しくは [WASM 入門](./なぜNodeでCを動かせるのか-WASM入門.md) の「檻」の話。

> **Q: C はどこで動くの？**
> A: native では機械語として Linux が直接。ブラウザと Node では **WASM に変換されて**動きます。
> **同じ C のソースが、3 つの場所すべてで（形を変えて）動いている**のがこの設計の肝です。

---

## 8. まとめ

1. **「書く言語」と「動く言語」は別物**。書くのは C / TypeScript、ランタイムに渡すのは
   機械語 / JavaScript / WASM。
2. **ただし最終的には全部、機械語になって CPU が走らせる**。違いは「いつ」——
   C は事前（AOT）、JavaScript / WASM は実行時（JIT）。WASM が速いのは型が明確で
   良い機械語になるから。
3. **TypeScript はどこでも実行されない**。走る前に必ず JavaScript になる（補助輪が外れる）。
4. **C は WASM（か機械語）になる**。ブラウザと Node では WASM として動く。
5. ブラウザも Node も、実行時は **「JavaScript（手）＋ WASM（脳）」の2層**で同じ構造。
6. `app/frontend` の TS → ブラウザ、`app/backend` の TS → Node、`app/shared` → 両方。
   **同じ TypeScript でも行き先が違う**。
7. 結局、**同じ C エンジンが 3 か所で動き、足りない"手"を JavaScript が補っている**——
   これが cub3D を Web に進化させる構成の全体像。
