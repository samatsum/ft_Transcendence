# ループ反転 — 誰がゲームを回しているか

**対象**: チームメンバー全員
**一言でいうと**: **ゲームを「回す人」を、環境ごとに差し替えられるようにした**話です。
Linux ではライブラリが、ブラウザではブラウザが、サーバでは Node が回します。
この作りのおかげで、**ゲームのロジックそのものは 3 か所で 1 つのまま**でいられます。

前提として [pfプレフィックス関数について](./pfプレフィックス関数について.md) を先に読むと分かりやすいです。

---

## 1. 「ループ」とは何のことか

ゲームは、ものすごく速い**パラパラ漫画**です。1 秒間に 30〜60 回、次のことを繰り返します。

```
①入力を読む → ②状態を更新（移動・当たり判定・勝敗） → ③画面を描く → ①へ戻る
```

この「①へ戻る」をやり続ける仕組みが**ループ**です。
**誰かがこのループを回し続けないと、ゲームは 1 枚目で止まったまま**になります。

問題は、**この「回す人」が環境によって全然違う**ことです。

---

## 2. 元の cub3D はどうだったか

元の cub3D は、**回す人＝MiniLibX（MLX）**でした。`main()` の最後はこうです。

```c
// codes/srcs/common/main.c
pf_setup_hooks(&game);   // 「毎フレーム main_loop を呼んで」と MLX に予約
pf_loop(&game.window);   // ← ここで MLX に制御を渡す。★もう return してこない★
```

`pf_loop` の native 実装は、たった 1 行です。

```c
// platform_native.c
void pf_loop(t_window* window) { mlx_loop(window->ptr); }
```

`mlx_loop`（[codes/minilibx-linux/mlx_loop.c](../../codes/minilibx-linux/mlx_loop.c)）は、
**42 から配布された MLX ライブラリの中**にある無限ループです。中身を削ると:

```c
int mlx_loop(t_xvar *xvar)
{
	while (win_count(xvar) && !xvar->end_loop)   // ★これがループ本体
	{
		while (... XPending(xvar->display))       // 溜まった入力イベントを捌く
			mlx_int_param_event[ev.type](...);    //   → key_press など
		if (xvar->loop_hook)
			xvar->loop_hook(xvar->loop_param);    // ← やることが無いとき main_loop(game) を呼ぶ
	}
	return (0);
}
```

`loop_hook` に入っているのが、`pf_setup_hooks` で登録した `main_loop` です。
つまり **MLX が、暇なたびに `main_loop` を延々と呼ぶ**。これが元の構造でした。

> **ここが移植の壁だった**: `mlx_loop` は Linux の X11 専用です。
> ブラウザにも Node にも存在しません。
> そして「回す人」が MLX だと、**ゲームを動かすには必ず MLX が要る**ことになります。

---

## 3. 「反転」とは何をしたのか

やったことは一言でいうと、**「回す人」と「1 フレーム分の処理」を切り離した**ことです。

### Before（元の cub3D）

```
MLX（回す人） ──呼ぶ──▶ main_loop（1フレーム分の処理）
        ↑ この2つが癒着していて、引き剥がせない
```

`main_loop` は「MLX から呼ばれる」ことが前提の関数でした。
MLX が居ないと呼んでもらえない＝動かせません。

### After（反転後）

```
      1フレーム分の処理を、引数付きの「ただの関数」として公開する
                    game_step(game, dt)
                          ▲
        ┌─────────────────┼─────────────────┐
   MLX が呼ぶ          ブラウザが呼ぶ        Node が呼ぶ
   （native）           （web）            （server）
```

**「1 フレーム分の処理」を、誰からでも呼べる普通の関数にした。**
すると「回す人」は誰でもよくなります。これを **ループ反転** と呼んでいます
（制御の向きが「ライブラリ→自分のコード」から「自分のコード←いろんな駆動元」に
変わるので「反転」。E-04 の作業）。

---

## 4. 3 つの「回す人」

同じ `game_step` を、3 者が別々の方法で呼びます。

| 環境 | 回す人 | ループの正体 | 1 フレームで呼ぶもの |
|---|---|---|---|
| **native** | MLX | `mlx_loop` の `while`（C・ライブラリ内） | `main_loop` → `game_frame` → `game_step` + `render_frame` + `pf_present` |
| **web** | ブラウザ | `requestAnimationFrame`（JS） | `tick` → `web_render`（= `game_step` + `render_frame`）→ JS が Canvas へ転送 |
| **server** | Node | `setInterval`（JS。W-10 で実装予定） | `game_step` のみ（描画しない） |

**3 つとも「回す人」が違うのに、`game_step` は 1 つ**。これがこの設計の狙いです。

### web の実物（[web/gate1.js](../../web/gate1.js)）

```js
function tick(now) {
	const dt = Math.min((now - last) / 1000, 0.05);  // native の frame_delta に相当
	last = now;
	moduleRef._web_render(dt || (1 / 60));           // C を呼ぶ（game_step + render_frame）
	present(moduleRef);                              // JS が Canvas へ転送（後述）
	requestAnimationFrame(tick);                     // ★次のフレームで自分をまた呼ぶ
}
```

最後の `requestAnimationFrame(tick)` が、native の `mlx_loop` の `while` に相当します。
**「次に画面を描くタイミングで、また `tick` を呼んで」**とブラウザに予約する。
ブラウザはモニタのリフレッシュ（普通 60Hz）に合わせて呼び返すので、延々と回ります。

### server の実物（現状は [web/sim_demo/record.mjs](../../web/sim_demo/record.mjs)）

本番の Node サーバ（W-10）はまだ未実装ですが、**同じ構造の動く例**がデモにあります。

```js
let tick = 0;
while (tick < MAX_SECONDS * TICK_HZ) {       // TICK_HZ = 30
	driveExternalSeat(tick);                 // 外部（人間）入力を入れる
	const finished = M._game_step(game, 1 / TICK_HZ);   // ★同じ game_step を呼ぶ
	tick++;
	if (tick % 2 === 0) snapshots.push(takeSnapshot(tick));  // 状態を記録して配信
	if (finished) break;
}
```

ここには `render_frame` も `present` もありません。**サーバは描かない**からです
（[pf の資料](./pfプレフィックス関数について.md) の headless の話）。
本番ではこの `while` が `setInterval(..., 1000/30)` に変わるだけで、中身は同じです。

## 5. 「回す人」が違うと、描画の担当も変わる

native の `game_frame` は、1 フレームの中で `pf_present`（画面転送）まで C がやっていました。
**web はここが違います。** C は「絵をメモリに書く」ところで止め、
**Canvas に映すのは JS の仕事**にしました。

```c
// web_main.c ― C 側。pf_present を呼ばない
int web_render(double delta_time)
{
	game_step(&g_game, delta_time);   // 状態を進める（3環境共通）
	render_frame(&g_game);            // フレームバッファ（メモリ）に絵を描く
	return (1);                       // ← ここで return。転送はしない
}
```

```js
// gate1.js ― JS 側の present。これが web での「pf_present の中身」
function present(Module) {
	const ptr = Module._web_framebuffer_ptr();   // C が描いたバッファの場所を聞く
	const src = Module.HEAPU8;                    // WASM のメモリを直接読む
	for (...) {
		rgba[out]     = src[i + 2];   // B ┐ MLX は BGRA 並び、Canvas は RGBA なので
		rgba[out + 1] = src[i + 1];   // G │ ここで並び順を入れ替える
		rgba[out + 2] = src[i];       // R ┘
		rgba[out + 3] = 255;
	}
	ctx.putImageData(image, 0, 0);    // ← ここで初めて Canvas に絵が出る
}
```

**C は「メモリに書く」、JS は「Canvas に映す」**と役割が割れています。
だから `web_framebuffer_ptr()` のような「バッファの場所を教える関数」が要ります。
これは [pf の資料 §6](./pfプレフィックス関数について.md) で触れた、hminemur（F-06）が扱う配線です。

---

## 6. まとめ

1. **ループ = パラパラ漫画をめくり続ける仕組み**。誰かが回さないとゲームは止まる。
2. 元の cub3D は **MLX が回す人**だった。MLX は Linux 専用なので、そのままでは移植できない。
3. **「1 フレーム分の処理」を `game_step` という普通の関数に切り出した**（ループ反転／E-04）。
4. すると回す人は誰でもよくなる。**native=MLX / web=ブラウザ(rAF) / server=Node(setInterval)**。
5. **`game_step` は 3 環境で 1 つ**。回す人と描く担当だけが環境ごとに違う。
