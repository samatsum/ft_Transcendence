# なぜブラウザは並列描画しないのか（描画性能と COOP/COEP）

**対象**: チームメンバー全員（とくに hminemur / F-06）
**一言でいうと**: **やれば遅くなる要素があり、しかも要らなかった**から、です。
「ブラウザは CPU をうまく使えない」からではありません。

---

## 1. 何の話か

cub3D の描画は **レイキャスト**という方式です。画面を縦に細い列へ切り、
**1 列ずつ「壁までの距離」を計算**して、その距離から壁の高さを決めて描きます。
960 幅なら 960 本の列を毎フレーム計算します。

列と列は互いに独立なので、**複数の CPU コアで分担すれば速くなる**はずです。
実際 native 版はそうしています。

```c
// codes/srcs/common/engine/render/cast_columns.c
jobs[i].start = (cols * i) / n;      // 各スレッドに担当する列の範囲を割り当て
jobs[i].end   = (cols * (i + 1)) / n;
pthread_create(&tid[i], NULL, &cast_worker, &jobs[i]);  // n 個のスレッドで並列
```

列ごとに書き込むピクセルが重ならないので、**mutex（排他ロック）が 1 つもありません**。
並列化に理想的な構造です。

**ところが web 版は、これを使わず 1 スレッドで全列を描きます。** なぜか。

---

## 2. 理由その1 — ブラウザで並列化すると「代償」がつく

Emscripten で C の `pthread` をブラウザ向けにビルドすると、こう変換されます。

```
C の pthread      →  Web Worker（複数コアで動く。ここは問題ない）
スレッド間の共有メモリ →  SharedArrayBuffer   ← ★これが厄介
```

**SharedArrayBuffer（SAB）** が問題です。

2018 年の **Spectre / Meltdown**（CPU の投機実行の穴を突いて、本来読めない
メモリを盗み見る攻撃）で、SAB が**超高精度なタイマー**として悪用でき、
攻撃の部品になることが分かりました。そのためブラウザ各社は SAB を一度**全面無効化**し、
今は条件付きでしか使えません。

その条件が **cross-origin isolation** です。サーバが**全レスポンス**に次の 2 つの
ヘッダを付ける必要があります。

```
Cross-Origin-Opener-Policy:   same-origin
Cross-Origin-Embedder-Policy: require-corp
```

これを有効にすると、**そのページが読み込む外部リソース全部**（画像・フォント・
他オリジンの何か）にも、対応するヘッダが必要になります。**1 つでも欠けると、
そのリソースが読めなくなります**。

### 私たちにとって、これが何を意味するか

- nginx（W-15）の設定が、その分ややこしくなる
- **評価の拒否条件「コンソールに警告・エラーゼロ」を、原因の分かりにくい
  ヘッダ由来のエラーで落とすリスク**を抱える
- W-15 は残り日程で沼ると痛い

要するに、**並列化の見返り（速度）に対して、リスクとインフラの手間が見合わない**。

---

## 3. 理由その2 — そもそも速度が足りていた

E-13 で、シングルスレッドのまま実測しました（開発機・Node ヘッドレス・300 フレーム）。

| 内部解像度 | fps（fps_map） | fps（rsp_map） |
|---|---|---|
| **960×540（既定）** | **112.0** | **106.8** |

**60fps 要件に対して約 1.8 倍の余裕**があります。
評価機が開発機の半分の性能でも要件を満たす計算です。

つまり「並列化しないと 60fps に届かない」という状況ではありませんでした。
**面倒を持ち込んでまで速くする理由が、そもそも無かった**わけです。

（参考: native は並列なので 1920×1080 でも約 110fps 出ます。
**解像度を下げるノブが要るのは、シングルスレッドの web だけ**です。）

---

## 4. 足りなくなったら、まず解像度を下げる

重い評価機に当たった場合の対処は、**pthread 有効化ではなく内部解像度**です。
その口はもう実装済みです（E-13）。

```
gate1.html?res=848x480      ← URL で内部解像度を指定できる
```

`web_init(map, is_rsp, width, height)` の後半 2 引数がこれです。
**内部解像度を下げても、Canvas 側は CSS で画面いっぱいに引き伸ばす**ので、
プレイヤーから見たウィンドウの大きさは変わりません（少し粗くなるだけ）。

描画コストはほぼピクセル数に比例するので、848×480 は 960×540 の約 78% の負荷です。

### 対処の優先順位（この順で。逆にしないこと）

1. **内部解像度を 848×480 に下げる** — 設定変更だけ。**無料**。ノブは実装済み
2. それでも足りなければ **COOP/COEP を入れて pthread 有効化** — 必須要件を壊しかねない

**2 を先にやってはいけません。** 1 はコストゼロ、2 は「コンソール警告ゼロ」を
危険にさらします。

---

## 5. まとめ

1. web が単一スレッドなのは、**ブラウザが非力だからではない**（Web Worker で並列は可能）。
2. ブラウザで並列化 = **SharedArrayBuffer** = **COOP/COEP ヘッダが必須**
   = nginx が複雑化し、**コンソール警告ゼロ要件を脅かす**。
3. しかも E-13 で **960×540＝112fps**、**60fps の約 1.8 倍**あり、そもそも要らなかった。
4. 重い環境には **内部解像度を下げるノブ（実装済み）で対処**。pthread はその後。
5. 描画のコード自体は並列化しやすい構造（列分割・mutex なし）なので、
   **やろうと思えばいつでも有効化できる**。ただし動機が無い。
