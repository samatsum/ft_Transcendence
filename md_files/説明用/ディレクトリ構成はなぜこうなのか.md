# ディレクトリ構成はなぜこうなのか

**対象**: チームメンバー全員
**一言でいうと**: **「言語 × 担当レーン」で切っている**、という話です。
「コードだから 1 か所」ではなく、「C の世界」「TypeScript の世界」「共通の入力データ」
「インフラ」を分け、さらにツールが要求する物はルートに残しています。

---

## 1. いまのルート直下

```
app/        TypeScript の Web アプリ一式（backend / frontend / shared + tsconfig.base）
codes/      C エンジン本体（srcs / includes / minilibx / tests）
web/        C を WASM 化した成果物＋それを動かす JS（build / gate1.js / sim_demo）
maps/       .cub マップ（ゲームの入力データ）
textures/   画像素材（.cub が参照）
infra/      nginx・証明書・Dockerfile（docker/）＝デプロイまわり
md_files/   資料一式（この説明も含む）
screenshot/ 実行時のスクリーンショット出力先（中身は git 管理外）

Makefile              C のビルド入口
package.json /
package-lock.json     npm（TypeScript）の入口と依存ロック
docker-compose.yml    docker compose up（評価の単一コマンド）
README.md             プロジェクトの顔
```

（このほか `node_modules/`＝npm 生成物、`cub3D`＝ビルド出力、`.gitignore` 等の
隠しファイルがありますが、いずれも「自動生成」か「隠し」なので数に入れません。）

---

## 2. 切り分けの軸は「コードかどうか」ではない

「コードなら `codes/` に集めたほうが素直では?」と感じるかもしれません。
でも、実際の軸は **言語（ツールチェーン）** と **担当レーン** の 2 つです。

### 軸1: 言語（＝ビルドの仕組み）で分ける

`codes/` は「コード全般」ではなく、**C プロジェクト**です。
`Makefile`・MiniLibX・`.c`/`.h` の世界で、ビルドは `make`。
いっぽう `app/` は **TypeScript プロジェクト**で、ビルドは `npm`。

この 2 つは**ツールも依存関係も完全に別**です。同じフォルダに混ぜると、
lint やビルドの設定がぶつかります。**言語で切ると、ツールチェーンが素直に分かれます。**

| フォルダ | 言語 | ビルド |
|---|---|---|
| `codes/` | C | `make` |
| `app/` | TypeScript | `npm` |

### 軸2: 担当レーンで分ける

`app/` の中は、[6-チーム分担計画](../02_設計書/6-チーム分担計画.md)の担当そのままです。

```
app/backend/   torinoue（Auth / REST / DB）
app/frontend/  mamiyaza + hminemur（画面）
app/shared/    全員が使う zod 契約
codes/         samatsum（完成済み。もう触らない）
```

**フォルダを開けば「誰の担当か」が分かる**のが利点です。
4 人で並行作業する今、この対応が効きます。

---

## 3. なぜ `app/` にまとめたのか（ルートを減らす工夫）

もともと `backend/` `frontend/` `shared/` `tsconfig.base.json` は**ルート直下に
バラけて**いました。これらは「TypeScript の Web アプリ」という 1 つの塊なので、
`app/` 配下へまとめました。ルートから **3 ディレクトリ + 1 ファイルが消えて `app/` 1 つ**に。

担当レーンとの対応は `app/backend`（torinoue）のように**1 階層下で保たれる**ので、
「フォルダ＝担当」の見やすさは失っていません。

> **なぜ壊れないのか**: 各パッケージは互いを**パッケージ名**（`@ft/shared` など）で
> 参照しているので、フォルダを動かしても import は無傷です。
> `tsconfig.base.json` も一緒に `app/` へ下ろしたため、`extends: "../tsconfig.base.json"`
> の相対パスは変わらず解決します。実装コードは 1 行も変えていません。

---

## 4. なぜ「これはルートのまま」なのか

ルートに残っている物の多くは、**ツールがそこにあることを要求する**からです。
動かすと壊れるか、規約と戦うことになります。

| ルート固定の物 | 理由 |
|---|---|
| `package.json` / `package-lock.json` | npm workspaces のルート。ここでしか成立しない |
| `Makefile` | `make` の既定入口。動かすと `make -f codes/Makefile` になり不便 |
| `docker-compose.yml` | `docker compose up`（評価の単一コマンド）がルートを見る |
| `README.md` | 課題要件。プロジェクトの顔はルート |
| `node_modules/` | npm が生成。移動不可（git 管理外） |
| dotfiles（`.gitignore` 等） | 隠しファイル。通常の `ls` に出ない |

**この「床」より下にはルートを減らせません。** ここは諦めるのが正解です。

### 各設定ファイルは何をするか（JSON はコメントを書けないので、この表が代わり）

`package.json` や `tsconfig` は **JSON なので `//` コメントを書けません**（npm/tsc が弾く）。
そこで役割はここに一覧します。`package.json` 系は各ファイルの **`description` フィールド**にも
一行説明を入れてあります。

| ファイル | 役割 |
|---|---|
| `package.json`（ルート） | npm workspaces のルート定義。`app/*` をまとめ、`dev`/`build`/`typecheck` スクリプトを持つ |
| `package-lock.json` | 依存ライブラリのバージョン固定（**自動生成。手で編集しない**） |
| `app/tsconfig.base.json` | 全 TS パッケージ共通の tsc 設定。`app/*/tsconfig.json` が `extends` する |
| `Makefile` | C の 3 ターゲット（`make`=native / `web` / `sim`）と `test` / `check` のビルド定義 |
| `docker-compose.yml` | いまは `engine-build`（emcc ビルド環境）のみ。W-15 で `nginx` / `app` を追加 |
| `.env.example` | 環境変数のひな型（実体の `.env` は git 除外）。ポート等をここで宣言 |
| `.gitignore` / `.dockerignore` / `.gitattributes` | git / docker の除外リストと属性設定 |

各パッケージの `description`（＝コメント代わり）は、フォルダ単位では
`app/backend`・`app/frontend`・`app/shared`・`web`（CJS 固定 shim）にも入っています。
`web/` の全ファイルの役割は [web/README.md](../../web/README.md) にまとめてあります。

---

## 5. あえて動かさなかった物

| フォルダ | 動かさなかった理由 |
|---|---|
| `codes/` | C プロジェクトそのもの。今のままが正しい |
| `web/` | `Makefile`（`web/build`）・`record.mjs`・CI が参照。動かすとパス書き換えが多い。W-10 で backend が読みに来る時に見直すのが自然 |
| `maps/` `textures/` | **評価者が打つ `./cub3D maps/...` が変わってしまう**。`.cub` 内の `./textures/...` 全行と USER_DOC も直す羽目に。費用が大きく評価に触るので据え置き |

「コードを `codes/` へ」の発想でいくと `maps`/`textures` も入れたくなりますが、
**これは評価者が打つコマンドを変えてしまう**ので、あえて外に置いています。
`maps`/`textures` は「コード」ではなく「ゲームの入力データ」だ、という整理です。

---

## 6. まとめ

1. 切り分けの軸は「コードか否か」ではなく **言語（`make` か `npm` か）× 担当レーン**。
2. `codes/`＝C エンジン、`app/`＝TypeScript の Web アプリ、と**言語で分ける**とツールが素直。
3. `app/` は担当（backend=torinoue 等）が**1 階層下で分かる**。フォルダ＝担当。
4. `backend/frontend/shared` を `app/` にまとめ、ルートを 3 つ減らした。パッケージ名参照
   と tsconfig の相対パスのおかげで、**実装は 1 行も変えず**に移動できた。
5. `package.json` / `Makefile` / `docker-compose.yml` / `README` などは**ツールが要求する
   のでルート固定**。ここが最小の床。
6. `maps`/`textures`/`web` は据え置き。**評価者のコマンドやビルドのパス**に触るため。
