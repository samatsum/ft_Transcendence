### CI の実行状況（2026-07-23 追記）

「毎 PR で自動実行される」は満たしたが、**green は未達のまま E-14 を閉じてしまった**。実際には2つの問題があり、いずれも後から特定・修正した。

| ジョブ | 症状 | 原因と修正 |
|---|---|---|
| `web-app`（W-01 で追加） | ✅ success | — |
| `native` | `make` exit 2（**E-14 導入時から**） | minilibx のビルドは `test/mlx-test` も作り、Linux ではそのリンクが `-lbsd` を要求する（`codes/minilibx-linux/test/Makefile.mk`）。CI に **`libbsd-dev` が未導入**だった（README の必要パッケージには記載あり。開発機には導入済みのため再現しなかった）。→ 追加して **success 確認済み** |
| `wasm` | `record.mjs` exit 1（**W-01 以降**） | W-01 で追加したルート `package.json` の `"type": "module"` により、Node が `web/build/*.js`（emcc 生成の CommonJS glue）を ESM と誤認。`require()` の戻り値が関数でなくなり `TypeError: createSim is not a function`。→ `web/package.json` に `{"type":"commonjs"}` を置いて解消 |

**診断の記録**: ジョブログは API 403 で読めなかったため、失敗時に出力を `::error::` で annotation へ出す計測ステップを CI に足して原因行（`record.mjs:29`）を特定した。推測で直さず計測してから直したことで、当初の仮説（emcc 版差で決着時刻が変わる）が誤りだと分かった。

**教訓**:
1. 開発機に既に入っている依存（`libbsd-dev` 等）は、CI のクリーン環境で実際に green を見るまで欠落に気づけない。**E-14 の受入を「ワークフローが実行される」で閉じたのは早計**で、「green を確認する」まで含めるべきだった。
2. ルートに `"type": "module"` の `package.json` を置くと、リポジトリ内の既存 `.js` の解釈が変わる。**既存 Node ツールの再実行を受入確認に含める**こと。

## E-13: 解像度別スループット（開発機・Node ヘッドレス、300 フレーム）

計測: `node web/bench_render.mjs [maps/相対パス]`（再実行可能な形でコミット済み）

**fps_map/1.cub**

| 内部解像度 | ms/frame | fps |
|---|---|---|
| 1920×1080 | 35.23 | 28.4 |
| 1600×900 | 24.56 | 40.7 |
| 1280×720 | 15.80 | **63.3** |
| 960×540（既定） | 8.93 | **112.0** |
| 848×480（下限） | 7.09 | 141.0 |

**rsp_map/rsp.cub**（スプライト・ハンド描画を含む）

| 内部解像度 | ms/frame | fps |
|---|---|---|
| 1920×1080 | 48.29 | 20.7 |
| 1280×720 | 21.20 | 47.2 |
| 960×540（既定） | 9.36 | **106.8** |
| 848×480 | 9.90 | 101.1 |

- 描画コストはほぼピクセル数に比例（レイキャスト列数×行数が支配項）。**既定の 960×540 は 60fps 要件の約 1.8 倍の余裕**があり、評価機が開発機の半分の性能でも要件を満たす。
- さらに重い環境では 848×480 へ一段縮小できる（Canvas 側は CSS 拡大なので見た目のサイズは不変）。RSP の 848×480 が 960×540 と拮抗しているのは実行ごとの揺らぎの範囲。
- native は並列レンダラ（pthread）のため 1920×1080 でも約 8.5ms/frame（≒110fps）。解像度ノブが要るのは**シングルスレッドの web だけ**（① §4-D で web は pthread 無効の決定）。

### 使い方（F-06 への申し送り）

- `web_init(map_ptr, is_rsp, width, height)` — 0 で既定。実際の値は C 側が **848×480〜1920×1080 にクランプ**する（`MIN_WIDTH` 等は native と共有の定数のため、それ未満へ下げたい場合は設計判断が要る。現状の余裕では不要）。
- 手動確認: `gate1.html?map=...&res=1280x720`（フッターに実解像度と fps が出る）。
- F-06 の GameView は初期化時に一度選べばよい（動的切替は再 init が必要。必要になった時に検討）。

## E-14: CI 構成

| ジョブ | 内容 |
|---|---|
| native | ubuntu-latest。X11 ヘッダ導入 → `make -j` → `make check`（13 lint、復旧済み layering 含む）→ **`make test`（sim 受入 85 checks）** → `xvfb-run timeout 5` でホワイトリスト4マップ+1.cub の起動スモーク（rc=124=5秒完走が成功） |
| wasm | `emscripten/emsdk:latest` コンテナ（docker-compose.yml と同じ既定イメージ）。`make web sim` → `node web/sim_demo/record.mjs`（seed 固定で決着までの通し実行） |

- ビルドの exit code を CI が直接見るため、「emcc 不在で古い wasm のまま検証する」型の事故は構造的に再発しない。
- **W-16 への申し送り**: FE lint/型検査はこのワークフローへジョブ追加する（⑤ の E-14/W-16 統合決定）。emsdk イメージを固定したくなったら workflow の `container:` と compose の `EMSDK_IMAGE` を同時に上げること。
