## 0. 前提と今回確定した決定

| # | 論点 | 決定 | 理由 |
|---|---|---|---|
| D-11 | HUD の描画分担 | **C レンダラ（render.wasm）= 3D 世界 + 自分の手（`render_rsp_hand`）のみ。スコア・カウントダウン・対戦者状態・結果は React の DOM オーバーレイ** | 既存 C 資産を無改変で使い、スタイル調整とコンソールゼロ検証を DOM 側に寄せる。C 内蔵 UI（ミニマップ等 `FLAG_UI`）は web では既定 OFF（E-08 のキー割当で切替可として残す） |

## 1. ルートとガード

React Router による SPA。未認証で保護ルートへ来たら `/login` へリダイレクト（`GET /api/auth/me` で起動時判定）。

| path | 画面 | ガード |
|---|---|---|
| `/` | → `/lobby` へリダイレクト | 要認証 |
| `/login` / `/signup` | 認証 | 認証済みなら `/lobby` へ |
| `/lobby` | ロビー（§3.2） | 要認証 |
| `/game/:roomId` | ゲーム（§3.3） | 要認証 |
| `/profile/:id` | プロフィール（§3.4） | 要認証 |
| `/privacy` / `/terms` | Privacy Policy / ToS（§3.5） | **不要**（未ログインでも読める） |
| `*` | 404 | — |

## 2. 共通レイアウト

- **Header**: ロゴ（→ `/lobby`）、自分のアバター+名前（→ `/profile/:me`）、ログアウト。
- **Footer**: **Privacy Policy / Terms of Service へのリンクを全画面で常設**（拒否条件「フッターから到達可能」の担保。ゲーム画面でも match_end モーダルの外に表示される）。
- **Toast**: エラー・通知の共通表示枠（③ §1-A のエラーエンベロープ `msg` を表示、`code` は開発者コンソールに出さない — コンソールゼロ運用）。
- **ErrorBoundary**: 描画例外時に再読込導線を出す（白画面とコンソール未処理例外を防ぐ）。
- ブレークポイント: Tailwind 既定（モバイル < 768px / デスクトップ ≥ 1024px の2点を評価基準にする）。

## 3. 画面仕様

### 3.1 Login / Signup

- フィールドと検証は ③ §1-B の zod スキーマを**そのまま**使用（FE/BE 二重検証のデモ箇所）。エラーはフィールド直下に表示。
- 送信中は二重送信防止。401/409 はフィールドエラーへマップ（`email_taken` → email 欄など）。
- 成功 → `/lobby`。ここでロビー WS を接続する（§4）。

### 3.2 Lobby（本プロジェクトの中心画面）

データソース: REST（初期表示）+ ロビー WS（差分）。② §3 のメッセージと 1:1 で対応する。

| 領域 | 構成要素 | データ / アクション |
|---|---|---|
| クイックマッチ | RSP / FPS の2ボタン、参加後はキュー状態パネル（待機人数・自分の位置・**自動開始までの残秒**）。リーダーには「**AIを埋めて今すぐ開始**」ボタン | `queue_join` / `queue_leave` / `queue_fill_start`。表示は `queue_state`（`auto_fill_in_ms`, `is_leader`） |
| カスタムルーム | 「ルーム作成」→ rules フォーム（マップ選択 = `GET /api/maps`、先取点 3–21、速度倍率 0.5–2.0、AI 強さ。② §4-B の範囲を UI 化）/ 「コードで参加」入力（6文字） | `room_create` / `room_join`。ルームビューへ遷移 |
| ルームビュー | 座席カード×定員（アバター/名前/`is_ai` バッジ/ホスト冠）、rules 表示（ホストのみ編集可）、開始ボタン（ホストのみ）、ルームコードのコピー | `room_state` 全量再送をそのまま描画（差分管理しない）。`room_update_rules` / `room_start` / `room_leave` |
| フレンド | フレンド一覧 + presence バッジ（online/in_queue/in_game/offline）、申請の送信/承認/拒否 | 初期 = `GET /api/friends`、差分 = `presence_update`。申請操作は REST（③ §2-C）後に一覧再取得 |
| 試合フィード | 全体の試合結果がライブで流れるリスト（勝者チーム・プレイヤー名・モード） | `match_result` 受信で先頭に追加（**「リアルタイム更新が全接続ユーザーに反映」要件のデモ箇所**。評価台本でこの画面を並べて見せる） |

- `match_found` 受信 → トースト表示 + **自動で `/game/:roomId` へ遷移**（② §4-C。accept ダイアログは無い）。
- キュー参加中はルーム作成・参加ボタンを無効化（② §3-B の1ユーザー1コンテキスト原則を UI でも表現）。

### 3.3 GameView（`/game/:roomId`）

レイヤ構造（下から順）:

```
[Canvas 960x540 内部解像度・CSSレターボックス拡大]   ← render.wasm が描画（世界+自分の手）
[HUD オーバーレイ (DOM, pointer-events: none)]      ← スコア・状態・演出
[モーダル層 (countdown / match_end / 切断バナー)]    ← 操作を持つ
[Header/Footer は非表示、Footer リンクは match_end モーダル内に縮退表示]
```

| HUD 要素 | 内容 | データソース |
|---|---|---|
| スコアバー（上部中央） | RSP: `[赤 7 - 4 青]` チーム色付き。FPS: スコア非表示（レースなので進行状況のみ = 収集数 `x/y`） | `snapshot.match.score` / `world_delta` |
| 対戦者ステータス行（上部隅） | 各席: 名前 + 状態バッジ `connected / 切断中(残n秒) / AI` | `player_status` + `event(player_disconnected).grace_ms`（② §7-B の要求） |
| 自分の手 | **HUD には出さない**（C レンダラの `render_rsp_hand` が画面下部に描く）。`hand_changed`（自分）時は Canvas 縁のフラッシュのみ | `event(hand_changed)` |
| 得点演出 | `point_scored` で得点チームの色に画面縁をフラッシュ + スコアバー強調 | `event(point_scored)`（値の正本は snapshot 側 — 演出のみに使う。② §5-D） |
| カウントダウン | `3・2・1` の全画面オーバーレイ → `match_start` で消える | `event(countdown / match_start)` |
| match_end モーダル | 勝敗・最終スコア・各プレイヤーの成績（`GET /api/matches/:id`）・「ロビーへ戻る」 | `event(match_end).match_id` → REST |
| 自分の接続バナー | 自 WS 切断時「再接続中…」バナー + 自動再接続。復帰時 `welcome.resume=true` で「復帰しました」トースト | `useGameSocket` の状態機械（§4） |
| 観戦 HUD（保1） | `role=spectator` のとき視点切替ボタン（席1〜4）を追加 | `welcome.role`。切替はクライアント内で視点対象を変えるだけ（② §5-E） |

**入力キャプチャ**（E-08 の JS 側と接続する契約）:

- Canvas クリックでキャプチャ開始、`Esc` で解放（解放中は移動入力を送らない）。
- キャプチャ中は矢印キー等のブラウザ既定動作（スクロール）を `preventDefault`。
- keydown/keyup → 論理軸 → `input { seq, yaw, mv }` を 30Hz 送信（② §5-A。`hand` は送らない = D-14、`act` は 0 固定）。
- 回転（←→）はローカルで yaw に積分して即時反映（② の唯一の予測）。タブ非表示（`visibilitychange`）で全キー解放（キー押しっぱなし事故の防止）。

### 3.4 Profile（`/profile/:id`）

| 領域 | 内容 | データソース |
|---|---|---|
| ヘッダカード | アバター・名前・登録日・最終ログイン・presence バッジ | `GET /api/users/:id` |
| 統計カード | モード別 played / win / lose / abandon / **win_rate**（③ §2-D の導出規約どおりの値） | `GET /api/users/:id/stats` |
| 対戦履歴 | テーブル（日時・モード・マップ・結果・スコア・相手）。mode フィルタ + ページネーション。行クリックで試合詳細モーダル | `GET /api/matches?user_id=` / `GET /api/matches/:id` |
| 本人のみ | 名前変更・パスワード変更フォーム、アバターアップロード（プレビュー・2MB/形式エラーの即時表示） | `PATCH /api/users/me` / `PUT /api/users/me/avatar` |
| 他人のみ | フレンド申請/解除ボタン | ③ §2-C |

### 3.5 Privacy Policy / Terms of Service

**プレースホルダー禁止**（拒否条件）。実データフローに即した実文面を用意する。最低限含める章立て:

- **Privacy**: 収集するデータ（email・パスワードのハッシュ・表示名・アバター画像・試合履歴・セッション Cookie）/ 利用目的（認証・対戦・統計表示）/ Cookie の説明（セッション維持のみ、トラッキングなし）/ 保存場所と期間（自己ホスト SQLite、アカウント存続中）/ 第三者提供なし / **アカウント削除機能が無いこと**と削除依頼の連絡先（README 記載の運用者）。
- **Terms**: サービス内容（対戦ゲーム）/ 禁止事項（不正アクセス・チート・不快な表示名）/ 無保証 / 42 教育プロジェクトである旨。

## 4. WS フックの状態機械（クライアント責務）

② の接続規約を UI 側の実装契約に落とす。

- **`useLobbySocket`**: ログイン後に接続、ログアウト/401 で切断。切断時は 1s→2s→5s（上限）の指数バックオフで自動再接続し、ヘッダに「オフライン」バッジを表示。close `4004`（多重接続で置換）を受けたら**再接続せず**「別のタブで開かれました」画面を出す（リロードで奪回可能）。
- **`useGameSocket`**: `/game/:roomId` マウント時に接続 + `join` 送信、アンマウントで close 1000。切断時は在ゲーム中なら自動再接続 + `join`（② §7 の 30 秒猶予に乗る）。close `4002`（ルーム消滅）→ トースト + `/lobby` へ。
- 受信メッセージは zod で検証し、**検証失敗はコンソール error を出さず**開発ログ関数（本番 no-op）へ（コンソールゼロ運用）。
- 補間は ② §5-C の契約どおり（100ms 遅延・2 スナップショット線形補間・自分の yaw はローカル優先）。実装は E-12 の受け口 JS と共同で持つ（担当境界: 壱 = wasm 呼び出しと補間計算、参 = フック・ライフサイクル・HUD 反映）。

## 5. 共通コンポーネント（最小セット）

`Button / Input / FormField / Card / Modal / Toast / Badge(presence・チーム色) / Avatar / Table / Tabs / StatCard / SeatCard / CopyField(ルームコード)`。カスタムデザインシステムのモジュールは**宣言しない**ため、数を揃えることより一貫性（色・余白・フォーカスリング）を優先する。

## 6. 受入条件

1. 全画面遷移 + 対戦1試合を DevTools コンソールを開いたまま通し、**error / warning ゼロ**（React StrictMode 有効のまま）。
2. デスクトップ幅とモバイル幅（375px）の2サイズで全画面が崩れない。GameView はモバイル幅で注意書きが出る。
3. 未ログインで `/lobby` 直リンク → `/login` へ。ログイン後に元の URL へ復帰する。
4. `/privacy` `/terms` にフッターから2クリック以内で到達でき、実文面が表示される。
5. ロビーを2ブラウザで開き、片方の試合終了がもう片方の試合フィードに数秒内に現れる（`match_result`）。
6. 対戦中に相手タブを閉じると、自画面の対戦者ステータスが `切断中(残n秒)` → `AI` に遷移して見える（② §7-B のデモ）。
7. キーボードのみで認証→キュー参加→対戦開始まで到達できる（フォーカス可視）。
