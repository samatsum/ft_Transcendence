## 0. 前提と仮定

本書は以下を仮定して設計を確定する。仮定が崩れた場合は §10 のリスク対応に従う。

- **A1**: 評価は評価者のマシン（またはチームPC）上で `docker compose up`（単一コマンド）によりローカル起動して行う。パブリッククラウドへの常時デプロイは不要。HTTPS は自己署名証明書（mkcert 等）で成立させる。
- **A2**: チームは C言語 に強く、Web（TypeScript/React/Node）は中級以下。ただし AI 支援により Web 側の生産速度は大きく底上げできる。
- **A3**: 「RSP 2vs2 先取10点」は、異チーム接触じゃんけんの**勝利1回 = 勝者チームに1点**、どちらかのチームが10点に到達した時点で試合終了、と定義する。
- **A4**: 「FPS 1vs1 レース」は、同一マップに2人が同時スポーンし、**先にゴールセルに到達した方が勝ち**。既存の「収集アイテム→扉開放」をゴールへの関門として再利用し、既存の敵（M）は妨害ハザードとして機能する、と定義する。
- **A5**: cub3D の実装状態は DEV_DOC.md の記述と一致している（`t_game` ファサード、`main_loop` 毎フレーム司令塔、`common`/`fps`/`rsp` 3系統、`.cub` マップ、レンダリングはCPUレイキャスト→ピクセルバッファ書き込み）。

## 1. プロダクト概要

### 1.1 コンセプト

「42 の C プロジェクト（cub3D）をそのままブラウザ対戦プラットフォームに進化させた」ことを技術的な売りにする。レンダラ・物理・ゲームルール・AI は既存 C 資産を再利用し、Web レイヤ（認証・ロビー・マッチング・統計・チャット的要素）を新規に構築する。

### 1.2 提供する2つのゲーム（確定仕様）

| 項目 | RSP モード（メイン） | FPS モード（セカンド） |
|---|---|---|
| 形式 | **2 vs 2 チーム戦**（計4プレイヤー） | **1 vs 1 レース** |
| 勝利条件 | じゃんけん接触勝利でチームに+1点、**先取10点** | **先にゴールセル到達**（収集→扉開放を関門に再利用） |
| 既存資産との差分 | NPC 3体 → リモート人間3人（不足席は AI が代替）、チームスコア追加、勝利判定変更 | 敵=妨害ハザード化、ゴールセル概念の追加、同時2人スポーン |
| 該当モジュール | Web-based game / Remote players / **Multiplayer 3+**（4人） | **Another game**（履歴+マッチメイキング付き） |
| AI の役割 | 空席・切断席を既存じゃんけんAIが代替 → **AI opponent** モジュール | 1人プレイ時の対戦相手（レースAI）として拡張可 |

既存 RSP がすでに「1人間 + 3NPC の4戦闘員」構造である点が最大の資産。**「NPC の入力を、ネットワーク越しの人間入力に差し替える」**のが本設計の中核アイデアであり、Multiplayer 3+ / Remote players / AI opponent を同一機構で成立させる。

## 2. 技術選定（トレードオフと結論）

### 2.1 エンジン移植戦略（最重要決定）

| 案 | 概要 | 長所 | 短所 | 2週間での現実性 |
|---|---|---|---|---|
| **A. Emscripten で C→WASM 移植（採用）** | MiniLibX を薄いプラットフォーム層（フレームバッファ+入力+時間）に置換し、クライアントは WASM で描画。同じ C シムを**ヘッドレス WASM として Node サーバ内でも実行**し、サーバが唯一の正 (authoritative) | ロジック二重実装ゼロ。C資産（レンダラ/物理/AI/じゃんけんルール/.cubパーサ）を最大再利用。評価時の技術的訴求力が最大 | Emscripten の学習コスト。MLX依存の剥離リファクタが必要。ビルドパイプラインが増える | ◎ レンダラは「ピクセルバッファに書くだけ」の構造なので Canvas と相性が良い。移植可否は Day 2 のスパイクで判定 |
| B. TypeScript へフルリライト | レイキャスタとルールを TS で書き直す | ツールチェーン単純。Web開発者に馴染む | C資産を捨てる。レンダラ+AI+物理の再実装は品質リスク。 「cub3Dを進化させる」という物語性を失う | ○ AI支援なら可能だが、検証済みC実装を捨てるのは損失が大きい |
| C. サーバでCネイティブ実行+フレーム配信 | サーバがレンダリングし映像をストリーム | クライアント実装最小 | 帯域・遅延・スケールが非現実的。「Webアプリ」としての体裁も弱い | × 却下 |

> **結論: 案A。** ただし Day 2 終了時点を go/no-go ゲートとし、スパイク（「静的マップが Chrome の Canvas に描ける」）が失敗した場合のみ案B（クライアント描画のみTSリライト、ルールはC実装を仕様書として移植）へフォールバックする。

**案Aの2ビルドターゲット**:

| ターゲット | 含むもの | 除くもの | 実行環境 |
|---|---|---|---|
| `sim.wasm`（サーバ用） | .cubパース、移動/衝突、敵AI、RSP勝敗、スコア、FPSゴール判定 | レンダラ、テクスチャ、入力デバイス、UI | Node (Fastify) 内。固定ティックで唯一の正として実行 |
| `render.wasm`（クライアント用） | レイキャスタ、スプライト描画、UI描画、上記シムの表示用状態 | X11/MLX 一切 | ブラウザ。サーバのスナップショットを受けて描画 |

### 2.2 必要な C リファクタリング（実装しない。境界の定義のみ）

現行の `main_loop` は「入力→更新→衝突/戦闘→描画」を1関数で毎フレーム実行している（DEV_DOC §1）。これを以下の3層に分離する。**この分離が本プロジェクト最初の技術タスクであり、担当は Engine 役1名+AI。**

| 層 | 内容 | 由来 |
|---|---|---|
| **プラットフォーム層（新規・薄い）** | フレームバッファ提供、入力イベント供給、時間取得、テクスチャバイト列供給。MLX の `mlx_*` 呼び出しをすべてこの層の関数に置換 | 新規（Emscripten側は Canvas/`requestAnimationFrame`/KeyboardEvent に接続） |
| **シミュレーション層（`game_step`）** | `main_loop` から描画を除いた部分：入力適用→移動/衝突→敵AI→接触解決（じゃんけん/ゴール）→スコア。**複数プレイヤー入力の配列**を受け取れるよう拡張 | `loop.c` / `camera.c` / `collision.c` / `enemy*.c` / `rsp_*.c` |
| **描画層（`render_frame`）** | 表示用状態（全戦闘員の位置・手・スコア）を受けてフレームバッファに描画 | `screen.c` / `draw*.c` / `sprite*.c` / `rsp_weapon.c` |

付随する置換事項:

- **ビジーウェイトの除去**: FPS上限のためのビジーウェイト（DEV_DOC §1）は、ブラウザでは `requestAnimationFrame`、サーバでは `setInterval` ティックに置換。`time_mult` 機構はそのまま活かす。
- **X11キーコードの抽象化**: `keymap.h` を論理キー（前進/後退/ストレイフ/回転）の enum に寄せ、物理キー対応は各プラットフォーム層が持つ。
- **XPM アセット**: ビルド時に PNG/生RGBA へ事前変換するアセットパイプラインを用意（既存 `PythonCodes/` に変換スクリプト資産あり）。実行時 XPM パースは廃止。
- **マルチプレイヤー化**: `t_game` の「カメラ=プレイヤー」前提を「戦闘員リスト（人間 or AI）+ 視点は自分の戦闘員」に一般化。RSP の4戦闘員構造をそのまま器にする。
- **pthread 並列レンダラ**: Emscripten の pthread は SharedArrayBuffer + COOP/COEP ヘッダが必要で難度が上がるため、**初期はシングルスレッド描画に固定**。内部解像度 640×360〜960×540 で描画し CSS でアップスケール。足りない場合のみ検討。

### 2.3 ネットワークモデル

| 案 | 概要 | 長所 | 短所 | 判定 |
|---|---|---|---|---|
| **サーバ権威 + スナップショット配信（採用）** | サーバの `sim.wasm` が 30Hz で唯一の正を計算。クライアントは入力送信のみ、15–20Hz のスナップショットを受けて 100ms 補間で描画 | 実装が最も単純。チート耐性。決定論不要（floatの再現性問題を回避）。観戦=同じスナップショット購読で無料同然 | 入力遅延を感じうる | ◎ 4人+数NPCの全状態は1KB未満/回。差分圧縮も不要 |
| ロックステップ（決定論同期） | 全クライアントが同じシムを実行 | 帯域最小 | float決定論の担保が困難。1人でも遅いと全員止まる | × |
| クライアント権威 | 各自が自機を計算し申告 | 遅延ゼロ感 | チート・不整合。評価で突かれる | × |

遅延対策は最小限に絞る: **視点回転のみクライアント側で即時適用**（見た目の応答性に最も効く）。移動の予測/巻き戻しは実装しない（60ms未満のLAN/ローカル評価では不要）。

**切断・再接続（Remote players モジュールの要求事項）**: セッショントークンで猶予時間内の再接続を許可。RSP では切断席を**即座に AI が代替**し、復帰時に人間へ戻す（AI opponent モジュールと相互に補強し合う）。FPS 1vs1 は片方切断で猶予後に不戦勝。

**WSプロトコル（概念設計）**: `join / input(seq, keys, yaw) / snapshot(tick, combatants[], score) / event(point_scored, hand_changed, goal, match_end) / spectate` の5系統。バイナリ化はせず JSON で開始し、帯域が問題になった場合のみ ArrayBuffer 化（YAGNI）。

### 2.4 Web スタック

| レイヤ | 選定 | 対抗案 | 選定理由 |
|---|---|---|---|
| フロントエンド | **React + Vite + TypeScript（SPA）** | Next.js / SvelteKit | SSR 不要（SSRモジュールは選ばない）。SPA が最も単純で、React は subject の定義上フレームワークとして認められている。AI のコード生成品質も最高クラス |
| スタイリング | **Tailwind CSS** | Bootstrap | 必須要件の「CSSフレームワーク使用」を充足。コンポーネント志向と相性が良い |
| バックエンド | **Fastify + TypeScript** | NestJS / Express / Django | Node なら `sim.wasm` を直接実行できる（**この一点が決定打**）。Fastify は WS プラグインが公式にあり軽量。NestJS は2週間には過剰 |
| DB | **SQLite + Prisma（ORM）** | PostgreSQL | 評価はローカル単一ホスト（A1）。コンテナ1つ減り、バックアップ=ファイルコピー。Prisma で ORM Minor を充足し、将来 Postgres 移行も容易 |
| リアルタイム | **WebSocket（@fastify/websocket）** | Socket.IO | 生 WS で十分。Socket.IO の抽象化は今回不要 |
| 認証 | メール+パスワード（argon2id）+ **httpOnly Cookie の JWT/セッション** | — | 必須要件の最低線を確実に。OAuth/2FA は保険モジュール（§5） |
| 入力検証 | **zod スキーマを FE/BE で共有** | — | 「フロント/バック双方での検証」要件を単一定義で充足 |
| リバースプロキシ/TLS | **nginx（自己署名証明書で HTTPS 終端、静的配信、WS プロキシ）** | Caddy | 実績と情報量。バックエンド内部は平文で可（subject 許容） |
| コンテナ | **Docker Compose（`docker compose up` 単一コマンド）** | — | 必須要件。構成: `nginx` + `app`（Fastify+sim.wasm）+ ボリューム（SQLite/アバター） |

## 3. システムアーキテクチャ

### 3.1 全体構成

```
[Chrome]                          [Docker Compose]
┌────────────────────────┐   HTTPS   ┌─────────┐      ┌──────────────────────────┐
│ React SPA              │◄─────────►│  nginx   │─────►│ Fastify (Node/TS)        │
│  ├ Lobby/Profile/Stats │  REST+静的 │  TLS終端 │      │  ├ REST API (auth/users/ │
│  ├ Privacy/ToS         │           │  WS proxy│      │  │   matches/stats)      │
│  └ GameView            │◄─────────►│          │      │  ├ Matchmaking Queue     │
│     ├ render.wasm      │  WSS      └─────────┘      │  ├ WS Gateway            │
│     │ (Canvas描画)      │   input→                   │  └ GameRoom × N          │
│     └ 入力キャプチャ      │   ←snapshot                │     └ sim.wasm (30Hz tick│
└────────────────────────┘                            │        サーバ権威シム)      │
                                                      │  Prisma ─► SQLite (vol)  │
                                                      └──────────────────────────┘
```

- **GameRoom**: 1試合=1ルーム=1つの `sim.wasm` インスタンス。ルームは同時複数（マルチユーザー必須要件の根拠）。試合終了時に結果を Prisma 経由で永続化。
- **観戦（保険モジュール）**: GameRoom への読み取り専用 WS 購読。追加コストほぼゼロの設計にしておく。
- **AI 席**: GameRoom 内で入力供給者が「WS クライアント」か「既存じゃんけんAI（sim内蔵）」かの違いだけにする。

### 3.3 データベーススキーマ（概念設計）

必須要件「明確なスキーマと明確に定義されたリレーション」および README の Database Schema 章に対応。

| テーブル | 主要カラム | リレーション / 備考 |
|---|---|---|
| `User` | id, email(unique), passwordHash, displayName, avatarPath, createdAt, lastSeenAt | 認証・プロフィール・オンライン状態の基点 |
| `Friendship` | id, requesterId, addresseeId, status(pending/accepted) | User 2者間。Standard user management の友達+オンライン状態表示に使用 |
| `Match` | id, mode(RSP/FPS), mapName, settingsJson, startedAt, endedAt, winnerTeam? / winnerUserId? | 1試合1行。settingsJson にカスタマイズ内容（先取点数・速度等）を記録 |
| `MatchPlayer` | id, matchId, userId?(null=AI), isAi, team, slot, pointsScored, result(win/lose/draw/abandon) | Match–User の中間。**AI席も行として残す**（統計の整合性のため） |
| `Session` / `RefreshToken` | id, userId, tokenHash, expiresAt | 再接続猶予・ログイン維持 |
| （保険）`OAuthAccount` | provider, providerUserId, userId | OAuth Minor 採用時のみ |

統計（勝率・対戦履歴・リーダーボード）はテーブルを増やさず `Match`/`MatchPlayer` の集計クエリで導出する（二重管理を避ける）。

| 保4 | Prometheus + Grafana 監視 | Major 2 | compose にコンテナ2つ追加 + Fastify メトリクスエンドポイント。ゲーム本体と独立なので終盤でも着手可 |

### 4.4 依存関係・整合性チェック

- ゲーム必須系（AI opponent / Multiplayer 3+ / Another game / Customization / Spectator / Game stats）→ **#1 の RSP がゲーム第1号**として全て充足。
- Another game の「最初のゲームが必要」→ RSP が先、FPS が後（スケジュールもこの順）。
- SSR は選ばない（ICP 非互換の注意も無関係化）。高度なチャット機能は選ばない（前提の User interaction Major を選んでいないため）。
- 「C→WASM エンジン移植」を Modules of choice（カスタム Major）として**宣言することはしない**。ゲーム系モジュールの実現手段そのものであり、二重計上と指摘されるリスクがある。評価で余地があれば口頭アピールに留める。
