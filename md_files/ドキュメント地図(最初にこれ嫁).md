## 2. ディレクトリ構成

```
md_files/
├── ドキュメント地図(最初にこれ嫁).md   ← いまここ（全資料の入口）
├── 01_課題/                            要件（42 の課題文）
├── 02_設計書/                          0=全体 / 1〜5=詳細設計（①〜⑤）/ 6=チーム分担
├── 03_実装レポート/                    完了した C 側作業の記録・証跡
├── 04_エンジン資料/                    既存 cub3D の仕様（規約・開発・ユーザー）
└── 説明用/                             **チームへの口頭説明用**。仕組みを1テーマずつ解説
```

**「説明用/」について**: 設計書は「何をどう作るか」を決めるための資料なので、
初見のメンバーには前提が多すぎます。`説明用/` は逆に**前提を持たない人へ説明するため**の
資料を、1 ファイル 1 テーマで置く場所です。samatsum が書き、チーム説明の台本として使います。

読む順の目安: **① pf → ② ループ反転 → ③ 3つのビルドターゲット** で「なぜ 3 環境で
動くのか」が通し、**④ スナップショットと補間 → ⑤ 戦闘員統合**でオンライン対戦の芯が分かる。
**⑥** は描画性能の補足（余裕があれば）。

| # | ファイル | 何が分かるか |
|---|---|---|
| ① | [pfプレフィックス関数について](./説明用/pfプレフィックス関数について.md) | 同じ C のコードが Linux・ブラウザ・サーバの 3 か所で動く仕組み（`pf_*` 境界） |
| ② | [ループ反転-誰がゲームを回しているか](./説明用/ループ反転-誰がゲームを回しているか.md) | ゲームを回す人が環境ごとに違う理由（MLX / ブラウザ / Node）と `game_step` の切り出し |
| ③ | [3つのビルドターゲット](./説明用/3つのビルドターゲット.md) | `make` / `make web` / `make sim` / `make test` が何を作り、何のためにあるか |
| ④ | [スナップショットと補間](./説明用/スナップショットと補間.md) | サーバ権威モデル。ブラウザに勝敗判定が無い理由と、写真の補間（F-06 の芯） |
| ⑤ | [戦闘員統合-人もNPCも同じリスト](./説明用/戦闘員統合-人もNPCも同じリスト.md) | 人も NPC も同じ席にして、Multiplayer / Remote / AI を 1 機構で成立させた話 |
| ⑥ | [なぜブラウザは並列描画しないのか](./説明用/なぜブラウザは並列描画しないのか.md) | web が単一スレッドな理由（COOP/COEP のリスクと、そもそも速度が足りていた） |
